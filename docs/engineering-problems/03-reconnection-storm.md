# 重连风暴：大面积断连后的正反馈雪崩

标签: `connection`, `scale`

## 问题是什么

网关重启、网络闪断或 CDN 故障后，数万设备几乎同时发起重连 → 网关瞬间过载 → 处理变慢 → 更多请求超时 → 客户端再次断连 → 再次重连 → 正反馈循环。

## 典型场景

```
网关重启:
  50000 个设备在 2 秒内发起重连
  → 网关 CPU 飙升到 100%
  → 握手处理延迟从 50ms 升到 5s
  → 客户端这端 5s 没收到握手响应 → 超时 → 断开 → 再次重连
  → 加上第一次还没处理完的请求... 雪崩
```

移动端特有：后台切前台、WiFi ↔ 蜂窝切换、弱网心跳超时——这些事件的规模远大于传统 Web 服务的客户端重连。

## 通用分析思路

1. **区分"新连接"和"重连"**：已经协商过 session 的设备应该比新设备优先拿到连接。
2. **客户端不能都同时重试**：需要 jitter（随机抖动）。
3. **服务端需要自我保护**：即使客户端不配合，服务端也能识别异常并限流。
4. **分层防御**：客户端退避 + 服务端限流 + 负载均衡 + 扩容信号——单层不够。

核心原则：**不要假设客户端会配合你的设计**。移动端的网络栈和操作系统重连逻辑不在你的控制范围内。

## 当前项目方案

### 客户端退避（Spec 05 §6.1）

```
重连间隔（带 ±25% jitter）:
  第 1 次: 0.5-1s
  第 2 次: 1-2s
  第 3 次: 2-4s
  第 4 次: 4-8s
  第 5 次: 8-16s
  第 6 次+: 16-30s（封顶）
```

如果旧连接存活 > 5 分钟，断开后先尝试快速重连（`RECONNECT` opcode 带旧 session_id）→ 失败再退避。

### 服务端保护（Spec 05 §6.2）

- 连接限流：每个 IP ≤ 5 新连接/s，每个 user_id ≤ 2 新连接/s
- 优先级队列：重连请求（有效 session_id，TTL 内）> 新连接
- 同一 user_id 并发连接请求只处理第一个——其余返回 429
- Gateway 无状态：连接状态在进程内存，路由在 Redis，单节点过载时 LB 自然导流

代码：`livechat-server/internal/gateway/manager.go` — `HandleUpgrade()` 接入 `ConnectionLimiter`（每 IP ≤5/s、每 user ≤2/s，超限 HTTP 429 / ErrorFrame 4029）。客户端退避见 `reconnect.go`。

## 替代方案及取舍

| 方案 | 优点 | 缺点 |
|------|------|------|
| **客户端退避 + jitter**（当前） | 零服务端开销 | 仍然依赖客户端守规矩 |
| **服务端连接队列 + 令牌桶** | 服务端主导限速 | 增加延迟、调参复杂 |
| **一致性哈希 + 粘性路由** | 减少重连跨度 | 单点热点风险 |
| **长连接池预热** | 切换更快 | 浪费资源 |

## 踩坑记录

- Jitter 方向很重要：±25% 随机，不是固定延迟。同一 batch 的设备如果都用相同算法（比如都取封顶 30s），会在 30s 后再次同步重连，形成"谐波效应"。
