# Spec 12 - 监控告警与压测方案

## 1. 目标

建立聊天系统的可观测性与容量验证机制，使团队能识别性能瓶颈、定位故障链路并验证架构改动是否有效。本 spec 重点学习如何从“能跑”过渡到“可度量、可验证、可回滚”的工程系统。

## 2. 核心挑战

- 聊天系统故障通常表现为“局部异常、用户体感明显”。
- 长连接、消息队列、缓存和数据库问题会相互放大。
- 没有压测基线就无法评估容量和扩容收益。
- 监控需要覆盖正确性指标（重复、乱序），而不仅是可用性与延迟。

## 3. 指标字典与 SLI

### 3.1 可用性指标

| 指标 | SLI 定义 | SLO（学习参照） | 告警阈值 |
|------|----------|----------------|----------|
| 消息接收 API 可用性 | `1 - (5xx 请求数 / 总请求数)` | ≥ 99.9% | < 99.5% P1 |
| 长连接接入成功率 | `成功握手数 / 总连接尝试数` | ≥ 99.5% | < 99% P1 |
| 增量同步成功率 | `成功同步请求数 / 总同步请求数` | ≥ 99.9% | < 99.5% P1 |

### 3.2 时延指标

| 指标 | SLI 定义 | SLO（P95） | 告警阈值 |
|------|----------|-----------|----------|
| 文本消息接收确认延迟 | 从客户端发送 HTTP 请求到收到 200 | < 300ms | P95 > 500ms P2 |
| 在线消息投递延迟 | 从消息持久化到对端设备收到 WebSocket 帧 | < 800ms | P95 > 1.5s P1 |
| 离线同步首屏恢复时间 | 从登录到拉完首屏 100 条事件 | < 2s | P95 > 3s P2 |
| 扇出消费者处理延迟 | 从 outbox_events 写入到消费完成 | < 200ms | P95 > 500ms P2 |

### 3.3 正确性指标

| 指标 | SLI 定义 | SLO | 告警阈值 |
|------|----------|-----|----------|
| 消息重复展示率 | `重复展示消息数 / 总展示消息数` | < 0.01% | > 0.05% P0 |
| 会话内顺序错误率 | `顺序错误检测数 / 总会话消息数` | < 0.01% | > 0.05% P0 |
| 多端已读状态不一致率 | `不一致检测数 / 总已读事件数` | < 0.1% | > 0.5% P1 |

### 3.4 资源与业务指标

| 指标 | 用途 | 采集方式 |
|------|------|----------|
| 长连接数 | 容量规划 | 网关暴露 / Redis 路由键统计 |
| 在线用户数 | 活跃度与容量 | 网关去重统计 |
| 消息写入 QPS | 核心吞吐 | API 网关计数 |
| 扇出队列积压 | 消费者健康度 | `outbox_events` pending 计数 |
| 同步事件积压 | 同步服务健康度 | `sync_events` 最大 lag |
| 推送成功率 | 推送通道健康度 | APNs 响应统计 |
| 热点群数量 | 异常行为检测 | Redis `hot_group:*` 统计 |
| DB 连接池使用率 | 数据库健康度 | DB 指标 |
| Redis 内存使用率 | 缓存健康度 | Redis INFO |
| CPU / 内存 / 网络 IO | 节点健康度 | Node Exporter / 云监控 |

## 4. 统一追踪与日志

### 4.1 Trace ID 传递

```
Trace ID 生成位置：API Gateway / WebSocket 握手时

传递路径:
  Client → Gateway → Message Service → Outbox Consumer → Fanout Service → Gateway → Client
  
每个日志/指标点必须携带:
  - trace_id
  - span_id
  - parent_span_id
  - user_id（可选，需脱敏）
  - conversation_id
  - server_message_id
```

### 4.2 关键日志事件

| 日志点 | 级别 | 内容 |
|--------|------|------|
| 消息接收成功 | INFO | `trace_id`, `server_message_id`, `conversation_seq`, 耗时 |
| 消息接收幂等命中 | INFO | `client_message_id`, `server_message_id` |
| Outbox 消费失败 | ERROR | `trace_id`, `event_id`, `retry_count`, 错误信息 |
| 扇出投递失败 | WARN | `trace_id`, `target_user_id`, `gateway_node`, 错误 |
| 同步游标落后过多 | WARN | `user_id`, `last_event_seq`, `latest_event_seq` |
| 推送令牌失效 | WARN | `user_id`, `device_id`, APNs 错误码 |

## 5. 分层仪表盘

### 5.1 业务层仪表盘

- 消息接收 QPS / 延迟 P50/P95/P99
- 在线消息投递成功率 / 延迟
- 消息重复率 / 顺序错误率
- 多端已读不一致率
- 推送成功率

### 5.2 服务层仪表盘

- API Gateway：请求数、错误率、延迟、连接数
- Message Service：接收处理耗时、DB 写入耗时
- Outbox Consumer：消费速率、pending 数量、失败率
- Fanout Service：投递成功率、目标不在线比例
- Sync Service：同步请求数、同步延迟、游标落后分布
- Push Orchestrator：推送发送数、成功/失败/拒绝比例

### 5.3 基础设施层仪表盘

- PostgreSQL：QPS、连接池、慢查询、锁等待
- Redis：内存、命中率、连接数、命令延迟
- Kafka/RMQ：吞吐量、消费 lag、分区不均衡
- 对象存储：上传/下载 QPS、错误率

## 6. 压测方案

### 6.1 压测目标

| 场景 | 目标 | 关键指标 |
|------|------|----------|
| 登录与连接建立 | 验证网关承载 20 万连接能力 | 接入成功率、握手延迟、CPU/内存 |
| 文本消息写入 | 验证 1 万 msg/s 写入能力 | 接收延迟、DB 写入延迟、错误率 |
| 群消息热点扇出 | 验证 200 人群高并发场景 | 扇出延迟、pending 积压、DB CPU |
| 离线同步回补 | 验证游标落后后的补拉能力 | 首屏恢复时间、同步成功率 |
| 重连风暴模拟 | 验证网关限流与退避 | 重连成功率、服务端资源使用 |

### 6.2 压测脚本框架

```python
# load_test/send_message.py
import asyncio
import httpx
import websockets
import json
import time
import random

class ChatLoadTester:
    def __init__(self, base_url, ws_url, users):
        self.base_url = base_url
        self.ws_url = ws_url
        self.users = users
        self.metrics = {
            'send_count': 0,
            'send_latency': [],
            'delivery_latency': [],
            'errors': 0
        }
    
    async def user_session(self, user):
        # 1. 登录获取 token
        token = await self.login(user)
        
        # 2. 建立 WebSocket
        ws = await websockets.connect(
            f"{self.ws_url}?token={token}"
        )
        
        # 3. 发送消息并计时
        for i in range(100):
            start = time.time()
            client_msg_id = f"{user['id']}_{int(start * 1000)}_{i}"
            try:
                resp = await httpx.post(
                    f"{self.base_url}/v1/messages/send",
                    headers={"Authorization": f"Bearer {token}"},
                    json={
                        "client_message_id": client_msg_id,
                        "conversation_id": user["conversation_id"],
                        "message_type": "text",
                        "content": f"load_test_msg_{i}"
                    }
                )
                if resp.status_code == 200:
                    self.metrics['send_latency'].append(time.time() - start)
                    self.metrics['send_count'] += 1
                else:
                    self.metrics['errors'] += 1
            except Exception as e:
                self.metrics['errors'] += 1
            
            await asyncio.sleep(random.uniform(0.1, 1.0))
    
    async def run(self, concurrency=100):
        semaphore = asyncio.Semaphore(concurrency)
        tasks = []
        for user in self.users:
            tasks.append(self.user_session(user))
        await asyncio.gather(*tasks, return_exceptions=True)
        self.report()
    
    def report(self):
        latencies = sorted(self.metrics['send_latency'])
        p95 = latencies[int(len(latencies) * 0.95)] if latencies else 0
        print(f"Send count: {self.metrics['send_count']}")
        print(f"Errors: {self.metrics['errors']}")
        print(f"P95 send latency: {p95 * 1000:.2f} ms")
```

### 6.3 重连风暴压测

```python
# load_test/reconnect_storm.py
async def reconnect_storm(ws_url, tokens):
    """模拟网关重启后所有设备同时重连"""
    # 1. 先建立所有连接
    connections = []
    for token in tokens:
        ws = await websockets.connect(f"{ws_url}?token={token}")
        connections.append(ws)
    
    # 2. 同时断开
    await asyncio.gather(*[ws.close() for ws in connections])
    
    # 3. 同时重连，测量成功率和时间
    start = time.time()
    results = await asyncio.gather(
        *[websockets.connect(f"{ws_url}?token={token}") for token in tokens],
        return_exceptions=True
    )
    success = sum(1 for r in results if not isinstance(r, Exception))
    print(f"Reconnect success: {success}/{len(tokens)} in {time.time() - start:.2f}s")
```

### 6.4 压测执行流程

1. **基线压测**：在改动前执行，记录当前性能基线。
2. **目标压测**：针对本次改动涉及的链路执行。
3. **极限压测**：逐步提升负载直到系统出现错误，找到容量上限。
4. **恢复压测**：停止压测后观察系统是否恢复正常，检查是否有积压或资源泄漏。

## 7. 故障演练手册

### 7.1 演练场景

| 场景 | 注入方式 | 预期行为 | 验收标准 |
|------|----------|----------|----------|
| Redis 不可用 | 关闭 Redis 节点或网络隔离 | 网关无法路由新连接，已建立连接保持；投递降级为离线同步 | 无崩溃、有降级告警 |
| MQ 堵塞 | 暂停 Outbox Consumer | outbox_events pending 增长，消费恢复后追上 | 无消息丢失、最终一致 |
| DB 主库宕机 | 切换主从 | 写操作短暂失败，读操作继续；自动或手动切主 | RPO ≈ 0，RTO < 30s |
| 推送服务延迟 | 模拟 APNs 延迟 | 推送延迟增加，但消息仍可通过 WebSocket/sync 到达 | 消息可达性不受影响 |
| 单网关节点宕机 | 停止一个网关 Pod | 该节点连接断开，客户端退避重连到其他节点 | 重连成功率 > 99% |
| 热点群消息洪峰 | 200 人群每秒 100 条消息 | 触发热点群保护，限流/批量/独立资源池生效 | 系统不雪崩、延迟可控 |

### 7.2 演练执行步骤

1. **准备**：选择非高峰时段，通知相关人员，准备回滚方案。
2. **注入**：按手册执行故障注入，记录开始时间。
3. **观察**：监控业务指标、资源指标、错误日志。
4. **恢复**：移除故障注入，观察系统恢复情况。
5. **复盘**：对比预期行为与实际行为，更新手册与架构。

## 8. 告警分级与响应

| 级别 | 条件示例 | 响应时间 | 响应动作 |
|------|----------|----------|----------|
| P0 | 消息重复率 > 0.05%、服务整体不可用 | 5 分钟 | 立即回滚或止血 |
| P1 | 在线投递延迟 P95 > 1.5s、接入成功率 < 99% | 15 分钟 | 排查并扩容/限流 |
| P2 | 单节点 CPU > 85%、Outbox 消费延迟 > 500ms | 30 分钟 | 观察并优化 |
| P3 | 容量接近阈值、非关键指标异常 | 次日 | 排期优化 |

## 9. 流程描述

每次架构变更后，先通过监控对照历史基线，再执行目标链路压测，最后结合日志和追踪数据分析瓶颈位置。没有基线、没有追踪、没有演练的系统无法证明自己具备扩展能力。
