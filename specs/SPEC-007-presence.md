# SPEC-007 — 在线状态 / 已读回执 / 正在输入：状态广播风暴治理

> 状态: Draft | Milestone: M2 | 依赖: SPEC-002, 003 | 被依赖: 008(离线判定)

## 1. 背景与动机（Why）

反直觉的事实：**状态类流量（presence/typing/receipts）在成熟 IM 里的量级
往往超过消息本身。** 消息是 1 发 N 收，而"我上线了"理论上要通知我的所有
联系人（N×M 问题），"正在输入"每次按键都可能产生事件。这个 spec 学的是
**如何对不重要的数据做降级设计**——与 SPEC-003 恰好相反的哲学。

## 2. 核心挑战与典型解法

### 挑战 A：Presence 的 N×M 广播风暴

1w 在线用户 × 平均 200 联系人 = 一次全员重连（网关重启后）产生 200w 次
presence 通知。天真的"上线即广播给所有联系人"方案在这里死亡。

**解法：从推模型改成订阅模型（Facebook/WhatsApp 的公开做法）：**
- 在线状态存 Redis：`presence:{uid} = {status, last_seen, gateway_id}`，
  由网关连接生命周期维护（连接建立置 online，断开置 offline + last_seen；
  TTL 90s 兜底网关崩溃没来得及清理的脏状态）；
- 客户端**只订阅当前屏幕需要的状态**：进入会话页 → `SubscribePresence(uids)`，
  离开 → 取消订阅。订阅关系存网关本地（连接死了订阅自然死）；
- 状态变更只推给订阅者集合，且**同一 uid 的变更 5s 内合并只推一次**
  （防抖：地铁里的手机每 30s 掉线重连一次，不能广播 N 次）。

### 挑战 B：已读回执（read receipts）在群里的放大

1:1 已读很便宜；500 人群里"已读"是 500 读者 × 通知发送者 = 又一次 N×M。

**解法：**
- 已读的本质是**每用户每会话一个水位** `read_seq`，不是每消息一个事件。
  客户端进入会话/滚动到底部时上报 `MarkRead{conv_id, conv_seq}`，只升不降；
- 1:1：水位变更实时推给对方（双勾变蓝的瞬间体验值得这一条推送）；
- 群：**不推事件**。发送者打开"消息详情"时按需拉取
  `SELECT count(*) FROM conv_read_marks WHERE conv_id=? AND read_seq>=?`
  （微信"n 人已读"就是拉模型）；
- 上报本身批量化：客户端 2s 窗口内多次滚动只上报最大水位一次。

### 挑战 C：Typing indicator —— 最完美的"可丢弃数据"

**解法（业界完全一致）：**
- 不落库、不进收件箱、不重试、不 ACK——纯内存转发，收不到就算了；
- 客户端节流：开始输入发一次，之后每 4s 至多一次；停止输入不发"停止"事件，
  接收端 6s 没收到新事件自动清除显示（超时即语义）；
- 只发给**正在看这个会话**的订阅者（复用挑战 A 的订阅关系），
  群里输入的人 > 3 时聚合为"多人正在输入"。

这三个挑战放在一个 spec 里，是因为它们共享同一条设计光谱——
**从"绝不能丢"（消息）到"丢了才对"（typing）**，学会给数据分级是本 spec 的灵魂。

### 挑战 D：last_seen 的隐私语义

WhatsApp 的经典产品-架构耦合点：last_seen 可见性设置（everyone/contacts/nobody）
必须在**服务端**过滤（客户端过滤 = 没过滤，抓包即泄露）。
读 presence 时叠加隐私策略，顺带学一次"权限检查必须在数据出口"。

## 3. 范围

**In**：presence 订阅系统、read_seq 水位链路（1:1 推 + 群拉）、typing 通道、
last_seen 隐私设置、iOS 端对应 UI（在线点、双勾变蓝、typing 气泡）。
**Out**：APNs 相关（008）、"最近在线"模糊化文案策略（backlog）。

## 4. 验收标准

1. 风暴实验：5w 连接全体断线重连（复用 SPEC-005 reconnect_storm），
   presence 相关 Redis ops 与推送帧数有防抖上界（指标证明合并生效——
   对比关闭防抖的对照组数据）。
2. 订阅正确性：A 在会话页看 B → B 掉线 → A 在 90s 内（TTL 兜底路径）或
   5s 内（正常断开路径）看到 B 离线；A 离开会话页后 B 的状态变更不再推送
   （抓帧证明）。
3. 群已读：500 人群发消息 → 全员已读 → 发送者的下行帧数为 O(1)（详情页
   拉取），而非 O(500)。
4. typing p99 < 200ms（同会话在线双方），且服务端 typing 路径零 DB 写入
   （代码审查 + 指标双重确认）。
5. last_seen 设为 nobody 后，任何 API/帧里都取不到该字段（安全测试断言）。

## 5. 测试计划

订阅表生命周期、水位单调性、防抖窗口单测；loadtest 增加 presence 混合场景；
隐私过滤做专项 API 测试矩阵（3 种设置 × 3 种关系）。
