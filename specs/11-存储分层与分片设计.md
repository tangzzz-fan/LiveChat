# Spec 11 - 存储分层与分片设计

## 1. 目标

定义聊天系统的数据分层、热点治理、缓存策略与水平扩展方案，使系统能从原型阶段平滑过渡到高并发场景。本 spec 重点学习如何根据访问模式选择存储介质、分片键与缓存策略。

## 2. 核心挑战

- 不同数据对象的读写模型差异极大：消息是写密集型，会话摘要是读密集型，在线状态是高频更新型。
- 热点群和热点会话会导致单分区压力失衡。
- 缓存策略不当会带来击穿、穿透和脏读问题。
- 分片策略必须与消息路由、同步游标设计保持一致，否则扩容会放大跨分片复杂度。

## 3. 数据分层

```
┌─────────────────────────────────────────────────────────────┐
│                        客户端层                              │
│  SQLite / CoreData / GRDB（本地消息、会话摘要、同步游标）      │
└─────────────────────────────────────────────────────────────┘
                              │
┌─────────────────────────────────────────────────────────────┐
│                        接入层                                │
│  Redis（在线路由、限流计数、热点标记）                         │
└─────────────────────────────────────────────────────────────┘
                              │
┌─────────────────────────────────────────────────────────────┐
│                        核心业务层                            │
│  PostgreSQL / MySQL（账号、设备、会话、消息、同步事件）         │
│  Kafka / RabbitMQ（Outbox 消费、媒体处理、推送编排）          │
└─────────────────────────────────────────────────────────────┘
                              │
┌─────────────────────────────────────────────────────────────┐
│                        基础设施层                            │
│  S3 / MinIO（媒体对象存储）                                   │
│  Elasticsearch / ClickHouse（日志、审计、可选）               │
└─────────────────────────────────────────────────────────────┘
```

| 数据类型 | 存储介质 | 原因 |
|----------|----------|------|
| 账号、设备、群关系 | 关系型数据库 | 强一致性、事务支持、复杂查询 |
| 消息主数据 | 关系型数据库（按 conversation_id 分片） | 会话内顺序查询、事务写入 |
| 同步事件流 | 关系型数据库（按 user_id 分片） | 按用户增量拉取、顺序消费 |
| 会话摘要 | 关系型数据库 + 缓存 | 高频读取、可事件驱动更新 |
| 在线路由 | Redis | 高并发、短 TTL、快速失效 |
| 限流计数 | Redis | 原子操作、过期自动清理 |
| 媒体文件 | 对象存储 | 大文件、低成本、高吞吐 |
| 审计日志 | 关系型数据库 / 时序库 | 可审计、可检索 |

## 4. 分片设计

### 4.1 分片键选择

| 表 / 数据 | 分片键 | 原因 | 与下游的对齐 |
|-----------|--------|------|-------------|
| `messages` | `conversation_id` | 会话内消息按 seq 范围查询 | 与 `conversation_seq` 单调递增假设一致 |
| `sync_events` | `user_id` | 每个用户独立消费自己的事件流 | 与同步 API `GET /sync/events?cursor=...` 一致 |
| `conversation_summaries` | `user_id` | 会话列表按用户查询 | 与 `sync_events` 同一分片键，便于联合查询 |
| `outbox_events` | `id`（按时间范围） | 消费者按时间顺序拉取 | 独立分片，消费端可并行 |
| `group_members` | `group_id` | 群成员操作按群聚合 | 与 `messages` 同分片键，减少跨分片 |
| `device_sessions` | `user_id` | 用户设备管理 | 与认证服务一致 |

### 4.2 分片函数

```
shard_id = hash(shard_key) % shard_count

# 或使用一致性哈希环，便于平滑扩容
shard_id = consistent_hash(shard_key, virtual_nodes=150)
```

**扩容策略**：
- 初始阶段：单库/单表 + 逻辑分片键，便于后续迁移。
- 扩容时：按 `conversation_id` 或 `user_id` 范围迁移数据，使用双写 + 校验保证一致性。
- 避免跨分片事务：同一业务操作尽量落在同一分片，如群消息写入 `messages` 与 `group_members` 都按 `group_id/conversation_id` 分片。

### 4.3 跨分片一致性

参考 Spec 06 §3.3：

- `messages` 按 `conversation_id` 分片，`sync_events` 按 `user_id` 分片。
- 群消息写入 `messages` 后，Outbox 消费者异步为每个成员写入 `sync_events`。
- 这两个写入不在同一事务中，属于最终一致。
- 客户端通过 `conversation_seq` 缺口检测兜底，必要时调用会话消息补拉接口。

## 5. 热点治理

### 5.1 热点群识别

```
判定条件（满足任一即标记为热点群，Redis TTL 60s）:
- 1 分钟内消息数 > 50 条
- 单消息扇出延迟 P99 > 1s
- 该群对应的 DB 分区 CPU > 70%

Redis Key: hot_group:{group_id}
Value: {detected_at, msg_rate, threshold}
```

### 5.2 热点群隔离策略

| 层级 | 策略 | 实现 |
|------|------|------|
| 写入限流 | 热点群消息写入限流 50 条/秒 | 网关层或消息服务层返回 429 |
| 独立资源池 | 热点群的 Outbox 事件进入独立消费者队列 | 避免阻塞普通群消息 |
| 独立分区 | 热点群 `messages` 迁移到独立物理分区 | 减少对其他群的 IO 影响 |
| 批量合并 | 在线投递与 sync_events 批量合并 | 500ms 一个批次 |
| 摘要去抖 | conversation_summary 每 2 秒批量更新一次 | 减少行级锁竞争 |

### 5.3 写扩散放大估算

详见 Spec 07 §4.2。以 200 人群、每秒 100 条消息为例，`sync_events` 与 `conversation_summaries` 的写入会被放大到约 2 万次/秒。这是分片与热点治理的核心驱动力。

## 6. 缓存策略

### 6.1 缓存分层

```
L1: 客户端本地缓存（SQLite + 内存）
L2: 服务端 Redis 缓存
L3: 关系型数据库（事实来源）
```

### 6.2 各数据缓存策略

| 数据 | 缓存键 | TTL | 失效策略 |
|------|--------|-----|----------|
| 会话摘要 | `conv_summary:{user_id}:{conversation_id}` | 5 分钟 | 事件驱动失效 + 写穿 |
| 用户在线状态 | `gateway:user:{user_id}:{device_id}` | 300 秒 | 心跳续期，断线删除 |
| 用户设备列表 | `devices:{user_id}` | 1 分钟 | 设备变更时失效 |
| 限流计数 | `rate_limit:{key}` | 1 分钟 | 自然过期 |
| 热点群标记 | `hot_group:{group_id}` | 60 秒 | 自然过期，重新判定 |

### 6.3 缓存问题防护

#### 缓存击穿

- 对热点键使用互斥锁（Redis SET NX），只允许一个请求回源。
- 对会话摘要等高并发读取场景，使用本地缓存 + Redis 二级缓存。

#### 缓存穿透

- 对不存在的会话/用户，缓存空值（NULL object），TTL 较短（如 30 秒）。
- 接口层做参数校验，避免非法 key 请求。

#### 缓存雪崩

- 热点键 TTL 加随机抖动，避免同时失效。
- 核心数据（如会话摘要）使用写穿策略，保证缓存与 DB 最终一致。

#### 缓存脏读

- 禁止缓存承担唯一可信写入责任，所有写操作以 DB 为准。
- 缓存更新采用“先写 DB，再删缓存”或“写穿（write-through）”策略。
- 读操作时若缓存未命中，从 DB 加载并回写缓存。

## 7. 容量与扩展规划

### 7.1 学习型规模假设（对应 Spec 01）

| 指标 | 数值 | 存储估算 |
|------|------|----------|
| DAU | 10–100 万 | — |
| 峰值在线连接 | 5–20 万 | Redis 路由表约 1–4 GB |
| 峰值消息写入 | 1k–10k 条/秒 | 日消息量约 1–10 亿条 |
| 单条消息平均大小 | 200 字节 | 日存储约 200 MB–2 GB（文本） |
| 媒体文件 | — | 对象存储，按容量线性扩展 |

### 7.2 扩展路径

1. **阶段 1**：单库单表，引入逻辑分片键（如 `conversation_id` 前缀）。
2. **阶段 2**：按分片键拆库拆表，引入 Redis 缓存与会话摘要投影。
3. **阶段 3**：热点群独立分区/资源池，Outbox 消费者分组消费。
4. **阶段 4**：跨地域部署、对象存储 CDN、全局一致性方案（P1）。

## 8. 流程描述

消息写入完成后，系统将消息主数据与摘要投影分开管理。主数据承担事实存储，摘要表承担高频读取优化，缓存进一步提升热点访问性能。分片策略需要与消息路由和同步游标设计保持一致，否则扩容后会放大跨分片复杂度。热点群通过限流、独立资源池和批量合并等手段保护系统稳定性。
