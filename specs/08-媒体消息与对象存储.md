# Spec 08 — 媒体消息与对象存储

## 1. 目标

定义图片、语音、视频和文件消息的完整生命周期：上传凭证签发、客户端直传、异步处理管线、下载授权和 CDN 分发。确保媒体链路不影响文本消息的主链路。

- **P0**：图片/视频/文件的分片上传、断点续传、缩略图生成、授权下载、媒体状态机。
- **P1 / 学习扩展**：视频转码、内容安全审核、CDN 边缘签名与全球分发、媒体 E2EE 加密。

## 2. 核心挑战

### 2.1 大文件上传与弱网的矛盾

- 几百 MB 的视频在 2G/3G 网上传可能耗时几分钟
- TCP 断开后必须从头重传（HTTP 本身不支持断点续传）
- 上传中切 WiFi/蜂窝 → 连接断开

### 2.2 异步处理管线

- 图片需要生成多种缩略图尺寸（P0）
- 视频需要转码 + 生成缩略图（P1，P0 可只存原视频/首帧封面）
- 内容安全审核需要异步完成（P1）
- 处理失败需要重试 + 状态回写（P0）

### 2.3 下载与消息链路的解耦

- 消息投递延迟必须是毫秒级，图片下载可以是秒级
- 接收方先收到消息元数据（含缩略图 URL），再按需/后台下载全尺寸

### 3.2 上传凭证（Presigned URL）

```
POST /v1/media/upload/initiate
{
  "mime_type":      "image/jpeg",

  "file_name":      "photo_20260709.jpg",
  "width":          1920,

Response:
{
  "upload_id":      "upload_xyz",
  "object_key":     "media/u_123/2026/07/09/img_abc123_original.jpg",
  "chunk_size":     5242880,       // 5MB 每分片
  "presigned_urls": [              // 分片预签名 URL
    "https://s3.../upload_xyz?part=1&sig=...",
    "https://s3.../upload_xyz?part=2&sig=..."
  ],
  "expires_at_ms":  1752685200000  // 15 分钟后过期
}
```

### 3.3 客户端分片上传

```
class ChunkedUploader {
    let chunkSize = 5 * 1024 * 1024  // 5MB
    let maxConcurrent = 3            // 最多 3 个并发分片
    
    func upload(file: Data, presignedUrls: [URL]) async throws -> String {
        // 1. 分片
        let chunks = file.split(by: chunkSize)
        
        // 2. 并发上传分片 (带进度回调)
        let etags = try await withThrowingTaskGroup { group in
            var results = [String?](repeating: nil, count: chunks.count)
            var queue = Array(chunks.enumerated())
            
            for _ in 0..<maxConcurrent {
                guard let (index, chunk) = queue.popFirst() else { break }
                group.addTask {
                    results[index] = try await self.uploadPart(
                        url: presignedUrls[index],
                        data: chunk,
                        index: index
                    )
                }
            }
            // ... 管理完成和后续分片
        }
        
        // 3. 通知服务端合并分片
        try await completeMultipartUpload(uploadId, etags, objectKey)
    }
    
    // 断点续传: 如果上传中断, 查询已完成的 parts
    // GET /v1/media/upload/{upload_id}/status
    // → {completed_parts: [1, 2, 3]}
    // → 从 part 4 继续
}
```

### 3.4 断点续传

```
GET /v1/media/upload/{upload_id}/status
→ {
  "upload_id": "upload_xyz",
  "object_key": "media/.../original.jpg",
  "status": "in_progress",
  "total_parts": 4,
  "completed_parts": [1, 2],    // 已完成的分片
  "expires_at_ms": 1752685200000
}

客户端从 part 3 继续上传。
无需重新上传已成功的分片。
```

## 4. 消息体元数据

### 4.1 Attachment 结构

```json
{
  "attachments": [
    {
      "attachment_id": "att_001",
      "object_key": "media/u_123/2026/07/09/img_abc123_original.jpg",
      "mime_type": "image/jpeg",
      "size_bytes": 2048000,
      "width": 1920,
      "height": 1080,
      "duration_sec": null,
      
      "thumbnail_key": "media/u_123/2026/07/09/img_abc123_thumb_320.jpg",
      "thumbnail_size": { "width": 320, "height": 180 },
      "thumbnail_encryption_key": "<encrypted_aes_key>",  // P1 E2EE 使用
      
      "encryption": {
        "algorithm": "AES-256-GCM",
        "encrypted_key": "<E2EE encrypted>",  // P1 用 Signal Protocol 加密
        "iv": "<base64>"
      },
      
      "blur_hash": "L6DqJ8_4%M%MIU%M_4t7?b%MM{Rj", // BlurHash 占位
      "upload_status": "complete",
      "moderation_status": "approved"  // P1 内容审核使用
    }
  ]
}
```

### 4.2 缩略图尺寸规范

```
图片:
  thumbnail_small:  120x120  (列表缩略图)
  thumbnail_medium: 320x320  (会话内预览)
  thumbnail_large:  800x800  (全屏预览第一阶段, 渐进式加载)

视频:
  thumbnail:        320x180  (视频封面)
  
生成策略:
  - 保持原始宽高比
  - JPEG 质量 80%
  - WebP 格式 (iOS 14+ 支持)
```

## 5. 异步处理管线

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│ Object Store │     │ Media Queue  │     │ Media Worker │
│ (S3 Event)   │────►│ (Kafka/RMQ)  │────►│ (Processor)  │
└──────────────┘     └──────────────┘     └──────┬───────┘
                                                  │
                    ┌──────────────────────────────┼──────┐
                    │                              │      │
               ┌────▼────┐  ┌──────────┐  ┌───────▼───┐  │
               │ Thumbnail│  │ Content  │  │ Metadata  │  │
               │ Generator│  │ Moderation│ │ Extractor │  │
               └─────────┘  └──────────┘  └───────────┘  │
                    │              │              │      │
                    └──────────────┼──────────────┘      │
                                   │                     │
                          ┌────────▼────────┐            │
                          │ Status Update   │            │
                          │ (Update Message │            │
                          │  .upload_status)│            │
                          └─────────────────┘            │
```

```
处理状态机:
  pending → processing → complete
                       → failed (retry → processing or failed_permanent)
  
  审核状态 (P1):
    pending → approved → 所有人可见
            → rejected → 仅发送者可见 "内容违规"
```

> P0 阶段：媒体处理管线只包含缩略图生成与元数据提取。内容审核与视频转码作为 P1 扩展，通过预留 `moderation_status` 字段和独立 worker 接入。

## 6. 下载与 CDN

### 6.1 授权下载

```
客户端请求下载链接:

POST /v1/media/download/auth
{
  "object_key": "media/u_123/2026/07/09/img_abc123_original.jpg",
  "conversation_id": "conv_abc"  // 校验: 用户是否在该会话中
}

Response:
{
  "download_url": "https://cdn.example.com/media/...?token=...&expires=3600",
  "expires_in_sec": 3600,
  "content_length": 2048000,
  "content_type": "image/jpeg"
}

CDN 配置 (P1 / 学习扩展):
  - CDN URL 带签名的 token (HMAC-SHA256)
  - Token 包含: object_key + user_id + expires_at
  - CDN Edge 校验 token 后回源或从缓存返回

> P0 阶段：下载链接直接回源对象存储（如 MinIO/S3），不做 CDN 边缘缓存与全球分发。P0 的下载授权仍通过服务端签名的临时 URL 实现。
```

### 6.2 渐进式图片加载（iOS 端）

```
策略:
  1. 消息到达后立即展示 BlurHash 占位图 (已在消息元数据中)
  2. 同时开始下载 thumbnail_small (120px) → 快速展示模糊版
  3. 下载 thumbnail_medium (320px) → 替换为清晰版
  4. 用户点击全屏 → 下载原图 (1920px)
  
实现:
  - 使用 Kingfisher / SDWebImage, 配置多级缓存
  - 内存缓存: 缩略图
  - 磁盘缓存: 原图 (LRU, 上限 200MB)
```

### 6.3 下载优先级

```
优先级排序:
  1. 当前可见会话的消息图片 (最高)
  2. 会话列表中前 5 个会话的图片
  3. 其余图片 (后台预下载, WiFi only)

iOS 端:
  使用 URLSession 的 discretionary 模式用于后台预下载
  可见会话使用 default 模式 (高优先级)
```

## 7. 媒体清理策略

```
自动清理:
  1. 阅后即焚媒体 → 到期后删除对象存储文件 + 标记消息（P1）
  2. 未发送成功的上传 → 24 小时后清理 upload_id 和分片（P0）
  3. 审核拒绝的媒体 → 7 天后清理（P1）
  4. 退群用户的群媒体 → 30 天后该用户的下载权限失效（P0）

垃圾回收:
  - 定时任务扫描 attachments 表
  - DELETE FROM attachments WHERE upload_status = 'orphan' 
    AND created_at < NOW() - INTERVAL '24 hours'
```
