# Spec 06 — 离线同步与多端一致性

## 1. 目标

定义设备离线恢复、历史补拉、增量事件同步和多端状态收敛机制。这是 WhatsApp「打开 App 就能看到最新消息」体验的技术基础，也是学习大规模实时聊天系统多端一致性的关键领域。

## 2. 核心挑战

### 2.1 离线消息缺口（Gap Detection）

设备离线期间对方可能发了 N 条消息。恢复时系统必须：
- 检测到消息缺口的存在
- 高效拉取缺失的消息（不能全量同步）
- 过程中不出现幽灵消息或顺序错乱

### 2.2 多端状态撕裂

同一账号的设备 A 和 B 同时标记会话为已读、同时发送消息——如果没有收敛规则，两端会话列表会永远不一致。

### 2.3 会话列表的单一真相来源

问题：会话列表的未读数、最后一条消息、排序——这些数据放在哪里？

- **方案 A：客户端实时计算：** 从消息表 GROUP BY + COUNT + MAX → 慢，离线消息不全时错误
- **方案 B：服务端预计算投影（推荐）：** `conversation_summaries` 表 + 增量更新
- **方案 C：客户端本地缓存 + 服务端校验：** 两端都要做

WhatsApp 使用服务端投影 + 客户端本地 DB 的双层架构。

## 3. 同步协议设计

### 3.1 同步事件流

```
全局事件序号 (global_event_seq):
  - 一个全局递增的 BIGINT 序列
  - 每个影响用户状态的操作产生一个事件
  - 事件写入 sync_events 表，按 user_id 分片

事件类型:
  event_type                payload
  ───────────              ───────
  message_created           {server_message_id, conversation_id, conversation_seq, ...}
  message_delivered         {server_message_id, user_id, delivered_at}
  message_read              {conversation_id, user_id, last_read_seq}
  conversation_updated      {conversation_id, last_message_preview, updated_at}
  group_member_added        {group_id, user_id, added_by}
  group_member_removed      {group_id, user_id, removed_by}
  device_added              {device_id, platform}
  device_revoked            {device_id}
```

### 3.2 SyncCursor 表

```sql
CREATE TABLE sync_cursors (
    user_id         BIGINT NOT NULL,
    device_id       VARCHAR(64) NOT NULL,
    last_event_seq  BIGINT NOT NULL DEFAULT 0,  -- 该设备已消费到的全局事件序号
    last_sync_at    TIMESTAMPTZ,
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    
    PRIMARY KEY (user_id, device_id)
);
```

### 3.3 SyncEvents 表

```sql
CREATE TABLE sync_events (
    event_seq       BIGSERIAL,                    -- 全局事件序号
    user_id         BIGINT NOT NULL,              -- 该事件的目标用户
    conversation_id VARCHAR(64),                   -- 所属会话 (可空)
    event_type      VARCHAR(50) NOT NULL,
    payload         JSONB NOT NULL,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    
    PRIMARY KEY (user_id, event_seq)              -- 按 user_id 分片
);

-- 索引：每个用户按序号倒序查询
CREATE INDEX idx_sync_events_user_seq 
    ON sync_events (user_id, event_seq DESC);
```

**注意：** `sync_events` 是按 `user_id` 分片的。这意味着一个群消息发送后，需要为**每个群成员**分别写入一条 sync_event。这是写扩散的代价，但对于增量同步的读取效率是值得的。

#### 跨分片一致性说明

- `messages` 表按 `conversation_id` 分片（见 Spec 11），`sync_events` 按 `user_id` 分片。
- 一条群消息写入 `messages` 后，Outbox 消费者为每个成员写入 `sync_events`。
- 这两个写入**不在同一事务**中，因此属于最终一致：成员可能在 `sync_events` 中先看到 `message_created` 事件，而对应 `conversation_seq` 的 `messages` 记录尚未同步到自己分片。
- 客户端通过 §4.4 的序号缺口检测兜底：若 `message_created` 事件中的 `conversation_seq` 与本地不连续，则调用会话消息补拉接口，直到缺口补齐。

### 3.4 同步 API

```
GET /v1/sync/events
Authorization: Bearer <access_token>
Parameters:
  cursor           = 0          # 上次同步到的 event_seq, 0 表示全量
  limit            = 100        # 最多返回 100 个事件
  include_messages = true       # 是否包含消息体(content)

Response:
{

    {
      "event_seq": 15001,
      "event_type": "message_created",
      "payload": {

        "conversation_seq": 42,
        "sender_user_id": "u_456",

```

## 4. 同步流程

### 4.1 冷启动同步（首次登录或游标过期）

```
Client                          Sync Service
  │                                   │
  │ GET /sync/events?cursor=0&lim=100 │
  │──────────────────────────────────►│
  │                                   │ 查询 sync_events
  │                                   │ WHERE user_id = $uid
  │                                   │ ORDER BY event_seq
  │                                   │ LIMIT 100
  │                                   │
  │ 返回 event_seq 1-100              │
  │◄──────────────────────────────────│
  │                                   │
  │ 客户端逐条应用到本地 DB             │
  │ 保存 cursor = 100                 │
  │ has_more = true → 继续拉取        │
  │                                   │
  │ GET /sync/events?cursor=100        │
  │──────────────────────────────────►│
  │ ...重复直到 has_more = false      │
```

### 4.2 在线增量同步

```
正常在线状态下:
  1. 新消息通过 WebSocket 实时投递 (MESSAGE_DELIVERY opcode)
  2. WebSocket 投递成功后，客户端更新本地 cursor
  3. 如果 WebSocket 投递失败（如连接抖动中）：
     → 消息仍会写入 sync_events
     → 连接恢复后客户端基于 cursor 补拉
```

### 4.3 断线恢复同步

```
客户端重连后的检查流程:

1. WebSocket 连接建立
2. 比较本地 cursor 与服务端 latest_event_seq
3. 如果 local_cursor < latest_event_seq:
   → 发起增量同步请求
   → 拉取 local_cursor 之后的所有事件
4. 如果某些会话检测到序号缺口 (conversation_seq 不连续):
   → 触发局部会话重建 (见 §5)
```

### 4.4 序号缺口检测

```
客户端本地检查 (重建连接后):

FOR each active conversation:
    local_max_seq = SELECT MAX(conversation_seq) 
                    FROM messages 
                    WHERE conversation_id = $cid
    
    expected_seq = 从服务端同步的最新 conversation_seq
    
    IF local_max_seq < expected_seq:
        // 有缺口! 需要补拉该会话的缺失消息
        GET /v1/conversations/{cid}/messages
            ?from_seq={local_max_seq + 1}
            &limit=50
```

## 5. 多端一致性收敛规则

### 5.1 已读状态收敛 —— 取最大值

```
规则: last_read_seq = MAX(本地值, 同步事件中的值)

设备 A: 读到了 seq=100
设备 B: 读到了 seq=95
设备 C: 离线

事件流:
  1. A 上报 last_read_seq=100 → sync_event {type: message_read, seq: 100}
  2. B 收到事件 → 本地 last_read_seq = MAX(95, 100) = 100
  3. C 上线 → 同步拉取 → last_read_seq = 100
  4. 最终: ABC 都是 100 ✓
```

### 5.2 未读数收敛

```
服务端 ConversationSummary 是未读数的唯一计算源:

当 A 发送消息时:
  UPDATE conversation_summaries
  SET unread_count = unread_count + 1
  WHERE conversation_id = $cid AND user_id != $sender_id;

当 B 已读时:
  UPDATE conversation_summaries
  SET unread_count = 0
  WHERE conversation_id = $cid AND user_id = $reader_id;
  
  -- 通知 B 的所有其他设备:
  INSERT INTO sync_events (user_id, event_type, payload)
  VALUES ($reader_id, 'conversation_updated', 
    jsonb_build_object('conversation_id', $cid, 'unread_count', 0));
```

### 5.3 消息排序 —— 强制使用 conversation_seq

```
客户端消息列表渲染:
  SELECT * FROM messages
  WHERE conversation_id = $cid
  ORDER BY conversation_seq ASC  -- 永远按 seq，不按时间戳

原因:
  - server_received_at 可能有微秒级倒序 (跨服务器时钟偏差)
  - conversation_seq 是单一写点分配的，严格递增
  - 即使 P1 阶段引入 E2EE，服务端无法读内容，seq 的分配仍然可靠
```

### 5.4 乐观更新 → 确定性收敛

```
客户端发送消息时:
  1. 先生成本地消息 (status = queued, 临时 local_id)
  2. 立即在 UI 展示 (乐观更新)
  3. 服务端确认后:
     - 更新该消息的 server_message_id, conversation_seq, status = accepted
     - 本地消息从 "临时记录" 变为 "正式记录"
  4. 如果收到 sync_event 中包含同一条消息 (其他设备也发了):
     - UNIQUE(server_message_id) 保证不会重复插入
```

## 6. 同步事件的清理策略

```
sync_events 不能无限增长。

策略:
  1. TTL 保留: 保留最近 30 天的事件
  2. 游标过期: 如果用户的 last_event_seq < (最新 seq - 100万):
     → 拒绝增量同步, 返回 410 Gone
     → 客户端降级为全量重建 (重新拉取所有会话摘要 + 近期消息)
  3. 每日清理: DELETE FROM sync_events 
     WHERE created_at < NOW() - INTERVAL '30 days'
```

## 7. ConversationSummary 投影

```
会话列表查询 (高频操作):
  SELECT * FROM conversation_summaries
  WHERE user_id = $uid
  ORDER BY 
    is_pinned DESC,
    last_message_at DESC
  LIMIT 50;

响应时间目标: P95 < 50ms

该表由消息写入和回执处理流程维护:
  - 新消息 → 更新 last_message, unread_count+1
  - 已读回执 → unread_count = 0
  - 群名/头像变更 → 更新对应字段
  - 用户退群 → 标记 is_hidden = true

一致性: 最终一致。目标收敛窗口:
- 同一会话内的摘要与消息列表：在单条消息处理完成后 < 100ms 内一致。
- 跨设备摘要同步：依赖 `conversation_updated` 事件，通常在事件投递后 < 500ms 内一致。
- 如果摘要与消息列表短暂不一致，客户端在同步时通过对比消息表来修正：以 `messages` 中的 `conversation_seq` 和 `server_received_at` 为准，更新本地会话摘要。
```
