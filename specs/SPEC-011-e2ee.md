# SPEC-011 — 端到端加密：Signal Protocol

> 状态: Draft | Milestone: M3（最后一块） | 依赖: SPEC-003, 009, 010 | 被依赖: 无

## 1. 背景与动机（Why）

E2EE 的威胁模型：**服务器本身不可信**（被黑、被迫交出数据、内部人员）。
目标：服务器只见密文，且做到前向保密（今天的密钥泄露解不开昨天的消息）与
后向保密（泄露后未来的消息也会重新安全）。WhatsApp 2016 年全量上线的就是
Signal Protocol，这是本领域唯一值得学的答案。

**为什么排最后**：加密会**冻结你的架构**——上线后服务端再也看不到内容，
所有依赖明文的功能（搜索、内容审核、服务端秒传去重、新设备历史回填路线 A）
同时失效或必须重设计。先在明文阶段把系统调通调稳，再上锁，是唯一明智的顺序。

## 2. 核心挑战与典型解法

### 挑战 A：双方从未同时在线，怎么交换密钥？—— X3DH

Diffie-Hellman 要求双方在线各出一半。IM 里 Alice 半夜给离线的 Bob 发首条消息。

**解法：X3DH（Extended Triple Diffie-Hellman）—— 预铸钥匙（prekeys）：**
- 每设备注册时上传：身份公钥 IK（长期）、签名预共享钥 SPK（中期，IK 签名，
  定期轮换）、一次性预共享钥 OPK × 100（用一个少一个，服务端发一个删一个）；
- Alice 发首条消息：找服务器要 Bob 的 prekey bundle → 本地做 3~4 组 DH
  （IK/EK × IK/SPK/OPK 组合）→ 派生初始根密钥 → 直接开始加密发送；
- 服务器在这里只是**公钥公告板**，看不到任何私钥或共享密钥；
- **OPK 耗尽的完整连动（评审 A4）**——三方各有明确职责：
  - Key Server：OPK 存量为 0 时照常返回 bundle（无 OPK 版本）并置
    `opk_exhausted=true`；每次取 bundle 消耗一个 OPK（原子 pop，防并发
    双发同一 OPK）；对同一目标的 bundle 请求限速（防恶意抽干 OPK 池）；
  - Bob（被取方）：每次上线 + 每收到 `opk_low` 服务端信令（存量 ≤ 20 触发）
    时补铸至 100 个，不等用完才补；
  - Alice（发起方）：收到 `opk_exhausted=true` 时正常走 SPK-only X3DH
    （牺牲该会话首消息的一次性前向性，协议允许），不报错不重试。

### 挑战 B：长会话的前向/后向保密 —— Double Ratchet

一把会话密钥用到死 = 泄露一次全历史沦陷。

**解法：双棘轮**——两个只进不退的"棘轮"叠加：
- **对称棘轮**：每条消息都从链密钥 KDF 出新消息密钥，用完即弃
  （前向保密：旧消息密钥不可逆推）；
- **DH 棘轮**：每次收发方向切换时混入新一轮 DH 输出，重置链
  （后向保密/自愈：泄露点之后的会话重新安全）；
- 乱序与丢包：跳过的消息密钥暂存（skipped keys，带上限防 DoS），
  与 SPEC-003 的 conv_seq 有序交付配合，实际 skip 很少发生。

### 挑战 C：群聊加密 —— Sender Keys

500 人群若用 500 个 pairwise 会话加密同一条消息 = 每条消息加密 500 次。

**解法：Sender Keys（WhatsApp 群方案）**：每个成员为本群生成自己的
"发送链密钥+签名密钥"，通过已有的 pairwise 安全信道分发给全体成员（一次性
成本 N 次，之后每条消息只加密一次广播）。代价与学习点：**成员变动 = 全群
重新分发**（踢人后必须换钥，否则被踢者还能解密）——这就是为什么群成员变更
在大群里"贵"，以及 SPEC-006"新成员看不到历史"在密码学层面的必然性。

### 挑战 D：与既有系统的碰撞清单（加密的真实成本）

| 功能 | 碰撞 | 处理 |
|------|------|------|
| 收件箱/游标 (003) | 无碰撞——可靠性机制只碰信封不碰内容 | 佐证 M1 分层正确 |
| 媒体 (009) | 服务端 sha256 秒传失效（密文各不相同） | 附件用随机对称钥加密后上传，钥+摘要放进加密消息体；去重降级为同设备本地去重 |
| 推送 (008) | 推送 payload 不能含明文 | payload 只带 conv_id/msg_id，Notification Service Extension 本地解密渲染 |
| 新设备历史 (010) | 路线 A（服务端拉）死亡 | 切路线 B：老设备加密打包经服务器中转 |
| 多设备 (010) | 会话是**设备对**级别，N×M 个 ratchet 状态 | Sesame 算法思想：per-device session 管理 + 设备变更事件驱动重建 |
| 身份验证 | 中间人：服务器可以偷换公钥 | Safety Number（双方身份钥指纹）UI + 变更告警，扫码比对 |

### 挑战 E：库选型与密码学纪律

**绝不手写密码学原语。** 选项：`libsignal`（Rust 核心 + Swift/Java 绑定，
Signal 官方，licence AGPL——学习项目无碍）。Go 服务端**不需要**任何解密能力
（这正是重点），只需存取 prekey bundle 与转发密文。
纪律：所有随机数走 CSPRNG；密钥材料不落日志；本地 ratchet 状态加密存
Keychain/GRDB（SQLCipher 可选加分项）。

## 3. 范围

**In**：Key Server（prekey bundle 存取 API，含 OPK 计数与防枚举限速）、
libsignal 集成、1:1 双棘轮全链路、群 Sender Keys、附件加密、加密推送渲染、
新设备历史转移（路线 B）、Safety Number UI、`content` 字段全面密文化。
**Out**：可否认性/元数据隐藏（sealed sender，文档一节讨论即可）、
后量子混合（PQXDH，文档提及）、密钥托管备份（backlog）。

## 4. 验收标准

1. **服务器盲测（本 spec 的灵魂实验）**：全量跑通 M1/M2 全部功能后，
   dump PG + Redis + 网关内存/日志，grep 任何一条测试消息的明文 → 零命中；
   messages.content 全部为密文（自动化脚本断言）。
   **豁免声明（评审 A12）**：bot 会话（013）架构性不加密，盲测脚本显式
   过滤 `WHERE conv_id NOT IN (bot 会话)`，豁免清单在测试用例中明文列出，
   禁止静默跳过；并加**反向断言**：bot 会话的 content 必须是明文
   （防止哪天 bot 会话被误加密，aisvc 静默读不懂而无人察觉）。
2. 前向保密实验：导出接收方当前 ratchet 状态 → 尝试解密该会话 100 条前的
   历史密文 → 失败。**"失败"的判定语义（评审 E5）**：libsignal 对无对应
   消息密钥的密文抛 `DuplicateMessageError`/`InvalidMessageException`
   （不会静默返回垃圾明文——AEAD 认证标签保证）；验收脚本断言"捕获到
   解密异常"，实施前先跑一个 spike 确认绑定层的具体异常类型并写进脚本。
2b. OPK 耗尽实验：抽干 Bob 的 OPK 池 → Alice 发首条消息成功（SPK-only
   路径），延迟 < 2× 正常首消息；Bob 上线后 OPK 池自动回补至 100
   （Key Server 计数为证）。
3. 踢人实验：500 人群踢掉成员 X → X 用被踢前的密钥材料解密后续群消息 → 失败；
   全群重分发的密钥消息量 = O(N) 且被批量优化（指标）。
4. 加密附件：MinIO 上的对象无法用任何服务端信息解出；接收方正常显图。
5. 加密推送：锁屏横幅显示解密后的正文（Extension 路径），服务端 payload
   抓包无明文。
6. MITM 演练：服务端恶意替换 Bob 的 prekey bundle → Alice 端 Safety Number
   变更告警弹出（拦截成功的 UI 证明）。
7. 性能回归：加密开启后 1:1 E2E p99 劣化 < 20%（对比 M1 基线），
   500 人群发送延迟仍 < 1s。
