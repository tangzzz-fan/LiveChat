# WhatsApp 类聊天系统规格总览与实施规划

## 1. 文档目标

本文档用于定义一套面向**学习大规模高负载实时聊天处理技术**的规格文档集合。目标不是完成静态 UI 演示或商业产品，而是通过逐步构建一个具备真实系统属性的聊天系统，深入理解高并发、弱网恢复、多端同步、消息状态收敛、可观测性与安全边界等核心问题。

## 2. 系统范围

### 2.1 P0 范围

- 私聊
- 群聊
- 文本消息
- 图片消息
- 会话列表
- 送达与已读回执
- 离线同步
- 多设备同步
- 推送唤醒
- 基础监控与压测

### 2.2 暂不纳入 P0

- 音视频通话
- 支付体系
- 社区/频道
- 复杂机器人生态
- 生产级端到端加密全量实现

## 3. 核心设计原则

- 先定义消息生命周期，再设计接口与存储。
- 先建立单一可信状态源，再实现界面投影。
- 先解决正确性，再优化吞吐与成本。
- 先形成模块边界，再考虑服务拆分。
- 先保留扩展点，再做性能局部优化。

## 4. 架构分层

### 4.1 客户端层

负责本地数据库、消息状态机、离线发送队列、会话投影、推送处理和 UI 展示。

### 4.2 接入层

负责 API 鉴权、长连接接入、用户路由、限流、灰度控制和基础协议解析。

### 4.3 核心业务层

负责消息、会话、群组、回执、同步、推送编排、媒体元数据和安全策略。

### 4.4 基础设施层

负责数据库、缓存、消息队列、对象存储、监控、日志、追踪和审计。

## 5. 关键非功能目标

- 消息接收链路具备幂等能力。
- 会话内消息顺序可恢复、可校验。
- 离线回补支持断点续拉。
- 多端已读状态能够收敛。
- 接入层能够承受重连风暴。
- 故障发生时可被监控系统感知。

## 7. 推荐实施顺序

### 第一阶段（P0 核心）

- `01` 产品边界与 SLO
- `02` 领域模型与消息生命周期
- `04` 消息发送主链路
- `05` 长连接网关与实时投递
- `06` 离线同步与多端一致性

这一阶段必须把消息正确性做透，否则后续功能都会反复返工。

### 第二阶段（P0 扩展）

- `03` 账号认证与设备管理
- `07` 群聊模型与扇出策略
- `08` 媒体消息与对象存储（P0 部分：上传/下载/缩略图）
- `09` 推送通知与后台唤醒

这一阶段补齐用户可感知能力，形成完整产品闭环。

### 第三阶段（P0 收尾 / P1 学习扩展）

- `10` 安全体系与密钥管理（P0：基线；P1：E2EE）
- `11` 存储分层与分片设计
- `12` 监控告警与压测方案
- `13` iOS 客户端架构设计

这一阶段用于支撑规模化、演进性与工程质量。

## 8. 按能力闭环的分阶段里程碑

本计划不绑定固定周数，而以**能力闭环**为单位推进。每个阶段完成后应能独立验证一条核心链路。

### 阶段一：消息正确性骨架（P0）

- 完成系统边界确认与 SLO
- 完成核心领域模型与消息状态机
- 完成消息发送主链路（Outbox + 幂等）
- 完成长连接网关与实时投递
- 完成离线同步与多端一致性

### 阶段二：用户可感知能力（P0）

- 完成账号认证与设备管理
- 完成群聊模型与扇出策略
- 完成媒体消息上传/下载与缩略图
- 完成推送通知与后台唤醒

### 阶段三：规模化与工程质量（P0/P1 交界）

- 完成安全基线与密钥管理预留
- 完成存储分层和分片策略
- 完成监控告警与压测方案
- 完成 iOS 客户端架构设计

### 学习扩展（P1）

- 生产级端到端加密（E2EE）全量实现
- 联系人发现的哈希隐私匹配
- 媒体内容审核与视频转码管线
- CDN 边缘签名与全球分发
- Notification Service Extension 解密推送预览

## 9. 流程描述

系统建设流程遵循“规格先行、主链路优先、局部扩展、压测验证”的策略。先通过 `01` 到 `06` 建立消息系统的正确性骨架，再通过其他 spec 叠加群聊、媒体、推送与安全能力，最后通过监控、压测和客户端架构固化，将原型升级为可持续演进的工程系统。

## 10. 交付约束

- 所有后续实现必须与本目录的文档编号保持一致。
- 设计变更必须先更新相应 spec，再修改代码。
- 任何新增能力都必须明确归属到已有 spec，或新建编号文档。
