# Spec 03 — 账号认证、设备管理与联系人发现

## 1. 目标

建立用户身份体系、多设备会话管理与 Token 生命周期。

- **P0**：账号注册/登录、JWT + Refresh Token、设备会话管理、设备列表与吊销、基础风控审计。
- **P1 / 学习扩展**：基于通讯录哈希匹配的联系人发现、Companion 扫码登录、每设备独立 E2EE 密钥分发。

> 本 spec 以理解 WhatsApp 式账号模型为目的，但 P0 实现应优先保证认证与设备管理正确，E2EE 与联系人发现作为后续扩展点。

## 2. WhatsApp 的账号模型特点

WhatsApp 与传统应用的关键区别：

| 维度 | 传统 App | WhatsApp |
|------|----------|----------|
| 账号标识 | 邮箱/用户名 | 手机号（E.164 格式） |
| 验证方式 | 密码/OTP | 短信验证码 |
| 多设备 | Session Token 共享 | 每设备独立的 DeviceSession + 密钥 |
| 联系人 | 用户主动搜索/添加 | 通讯录自动匹配（哈希隐私保护） |
| 主设备 | 无 | 手机是主设备，桌面是 Companion |

## 3. 核心挑战

### 3.1 手机号作为身份的利弊

**利：**
- 零摩擦，不需要注册用户名/密码
- 通讯录天然就是好友列表
- 减少假账号（短信验证成本）

**弊：**
- 手机号会更换（WhatsApp 支持换号迁移）
- 隐私问题（手机号被暴露）
- 国际号码格式不一致（E.164 规范化）

### 3.2 多设备密钥分发（P1 / 学习扩展）

WhatsApp 的多设备架构依赖**每设备独立密钥**：
- 手机是主设备，持有身份密钥对（Identity Key Pair）
- 桌面/Web 是 Companion 设备，各自有独立密钥对
- Companion 登录需要主设备扫码授权
- 每条消息需为每台设备单独加密

这比简单的 Session Token 共享复杂得多，但这是 E2EE 的代价。

> P0 阶段：设备间不强制 E2EE 密钥分发，消息通过 TLS + 服务端中转即可。保留 `public_identity_key` 字段，但不做密钥校验逻辑。

### 3.3 联系人隐私（P1 / 学习扩展）

WhatsApp 的做法：
1. 客户端定期上传通讯录电话号码的**SHA-256 哈希**
2. 服务端比对哈希值，返回匹配的用户列表
3. 服务端不存储原始手机号——仅存哈希
4. 不在通讯录中的用户仍然可以发消息（通过手动输入号码）

> P0 阶段：用户通过手动输入对方手机号/ID 发起聊天，不实现通讯录自动匹配。

## 4. 登录与认证流程

### 4.1 注册/登录流程

```
┌──────────┐     ┌──────────────┐     ┌──────────┐
│  Client  │     │ Account Svc  │     │ SMS/OTP  │
└────┬─────┘     └──────┬───────┘     └────┬─────┘
     │                   │                  │
     │ POST /auth/request_code              │
     │ {phone: "+8613800138000"}            │
     │──────────────────►│                  │
     │                    │ Send OTP         │
     │                    │─────────────────►│
     │                    │   200 {retry_after: 30s}│
     │◄──────────────────│                  │
     │                    │                  │
     │ POST /auth/verify_code               │
     │ {phone, code, device_info}           │
     │──────────────────►│                  │
     │                    │ Verify OTP       │
     │                    │─────────────────►│
     │                    │◄───── OK ───────│
     │                    │                  │
     │                    │ Create/Find User │
     │                    │ Create DeviceSess│
     │                    │ Issue Token Pair │
     │                    │                  │
     │ 200 {access_token, refresh_token,     │
     │      user_id, device_id,             │
     │      expires_in: 3600}               │
     │◄──────────────────│                  │
```

### 4.2 Token 设计

```
access_token:
  - JWT (RS256 签名)
  - payload: {sub: user_id, did: device_id, iat, exp, jti}
  - TTL: 1 小时 (短周期)
  - 每次请求在 Authorization 头携带

refresh_token:
  - 256-bit 随机字符串 (opaque token)
  - 服务端存储 SHA-256 哈希
  - TTL: 30 天
  - 支持 Refresh Token Rotation（P0 可选，P1 建议实现）

device_session:
  - device_session_id
  - user_id
  - device_id (客户端生成, UUID)
  - platform (ios/android/desktop_web)
  - push_token (APNs/FCM token)
  - public_identity_key (预留字段，P1 E2EE 使用)
  - session_version (整数, 吊销时递增)
  - last_seen_at
  - created_at
```

### 4.3 Refresh Token Rotation（P0 可选 / P1 建议）

```
每次用 refresh_token 换新的 access_token 时:
  1. 校验 refresh_token 哈希
  2. 检查是否已被使用 (reuse detection)
  3. 若已使用 → 该用户所有 refresh_token 全部吊销 (泄露信号!)
  4. 若正常 → 标记旧的为已使用, 签发新的 refresh_token
```

这是 WhatsApp 级别的 Token 安全策略。P0 可以先用静态 refresh_token，P1 再引入旋转与重放检测。

## 5. 多设备会话管理

### 5.1 设备类型

| 类型 | 说明 | P0 登录方式 | P1 扩展 |
|------|------|------------|--------|
| Primary (手机) | 主设备 | 短信验证 | E2EE 身份密钥对 |
| Companion (桌面/Web) | 辅助设备 | 暂不实现（P1） | 主设备扫码授权 + 独立密钥对 |

### 5.2 Companion 设备登录流程（P1 / 学习扩展）

```
1. Companion 生成密钥对 → 展示 QR Code (含临时令牌)
2. Primary 扫描 QR → 用自己的身份密钥签名 Companion 的公钥
3. Primary → Server: "我为设备 X 授权，附上签名"
4. Server 验证签名 → 激活 Companion DeviceSession
5. Server → Companion: 返回 access_token + refresh_token
6. Server → Primary 的所有其他设备: 广播 "新设备加入" 安全事件
```

> P0 阶段仅支持单手机设备登录。多设备 Companion 登录作为 P1 扩展保留。

### 5.3 设备列表与吊销

```
GET /devices
→ [
    {device_id, platform, last_seen, is_current},
    ...
  ]

POST /devices/{device_id}/revoke
→ 吊销该设备的所有 token 和密钥
→ 广播 "设备已移除" 事件
→ 该设备下次请求时收到 401 + device_revoked
```

## 6. 联系人发现（P1 / 学习扩展）

### 6.1 通讯录上传（哈希方式）

```
客户端:
  1. 读取通讯录 (CNContactStore, 需用户授权)
  2. 所有号码标准化为 E.164
  3. 每个号码计算 SHA-256(trimmed_lowercase_number)
  4. 批量上传哈希列表

POST /contacts/sync
{
  "hashes": [
    "a7f1c...",   // SHA-256(+8613800138000)
    "b3e2d...",   // SHA-256(+8613900139000)
    ...
  ],
  "contacts_version": 3  // 增量同步
}

服务端:
  SELECT phone_hash, user_id, display_name, avatar_key
  FROM user_contacts
  WHERE phone_hash IN ($hashes)

→ 返回匹配列表:
{
  "matches": [
    {phone_hash: "a7f1c...", user_id: "u_123", display_name: "张三"},
    ...
  ],
  "contacts_version": 4
}
```

### 6.2 隐私保证

- 服务端**不存储原始手机号**，仅存 SHA-256 哈希（不可逆）
- 非用户（未注册）的手机号哈希**不在服务端存储**
- 用户可以选择不被通讯录发现（隐私设置）
- 服务端不记录「谁上传了谁的哈希」

### 6.3 Social Graph 存储

```
user_contact_graph:
  user_id_1 (FK → User)
  user_id_2 (FK → User)
  relationship: mutual_contacts | frequently_contacted | blocked
  last_interaction_at
  created_at

-- 当 A 和 B 都在对方通讯录中 → relationship = mutual_contacts
-- 用于优化：双方互为联系人时可降低某些安全限制
```

## 7. 风控基础

### 7.1 登录风控规则

| 规则 | 阈值 | 动作 |
|------|------|------|
| 单号码 OTP 请求频率 | > 3 次/小时 | 拒绝 1 小时 |
| 单 IP OTP 请求 | > 20 次/小时 | 拒绝 + 验证码 |
| 异地登录 | 国家/IP 跃变 | 强制 OTP + 通知用户 |
| Refresh Token 重放 | 检测到旋转 | 全设备 Token 吊销 |
| 新设备登录 | 首次登录 | 通知其他设备 |

### 7.2 审计事件

```
login_audit_events:
  event_id
  user_id
  event_type: login_success | login_failed | code_request | device_added |
              device_revoked | token_refreshed | token_replay_detected |
              security_alert
  ip_address (加密存储)
  user_agent
  device_id
  failure_reason
  created_at
```
