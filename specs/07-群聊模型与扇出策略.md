# Spec 07 — 群聊模型与扇出策略

## 1. 目标

定义群聊的成员、角色和权限模型，以及从 2 人小群到 200 人大群的消息投递策略。核心问题是：**当 1 条消息需要触达 N 个人时，系统成本是 O(N) 还是 O(1)？** 本 spec 重点学习写扩散/读扩散取舍、热点群保护与扇出降级。

## 2. 核心挑战

### 2.1 写扩散 vs 读扩散

这是群聊的核心架构取舍。WhatsApp 群聊限定 256 人（后扩至 1024），Telegram 支持 20 万人——上限不同的根源就是扇出策略不同。

| 策略 | 写入时 | 读取时 | 适用规模 | 存储成本 |
|------|--------|--------|----------|----------|
| **写扩散 (Inbox Model)** | 为每个成员创建一条消息副本/投递任务 | 直接读自己的收件箱 | ≤ 256 人 | O(N × M)，N=成员, M=消息 |
| **读扩散 (Outbox Model)** | 只存 1 份消息 | 每人拉取时再去群信箱读 | 任意规模 | O(M) |
| **混合扩散** | 在线成员写扩散，离线成员读扩散 | — | 中等 | 折中 |

### 2.2 WhatsApp 的模式：写扩散 + 256 上限

WhatsApp 选择写扩散，设计取舍：
- **利：** 读取极快（每个人的收件箱已就绪）、离线同步简单
- **弊：** 群人数有硬上限、群消息的写入放大了 N 倍
- **为什么选 256？** 因为 256 人的写扩散成本可控，且覆盖了 99% 的群聊场景

Telegram 选择读扩散，可以支持 20 万人的超级群，但代价是读取时需要去群信箱拉取。

### 2.3 热点群问题

当某个群在短时间内产生大量消息（如直播讨论），所有成员的收件箱/投递队列同时被更新：
- 数据库热点行（该群的 message counter 或 summary）
- 缓存热点键
- 投递队列暴涨

## 3. 群模型

### 3.1 数据模型

```sql
CREATE TABLE groups (
    id              VARCHAR(64) PRIMARY KEY,
    name            VARCHAR(255) NOT NULL,
    description     TEXT,
    avatar_key      VARCHAR(255),
    creator_user_id BIGINT NOT NULL,
    max_members     INT NOT NULL DEFAULT 200,
    current_members INT NOT NULL DEFAULT 0,
    is_archived     BOOLEAN NOT NULL DEFAULT FALSE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    version         INT NOT NULL DEFAULT 1  -- 乐观锁
);

CREATE TABLE group_members (
    group_id        VARCHAR(64) NOT NULL,
    user_id         BIGINT NOT NULL,
    role            VARCHAR(20) NOT NULL DEFAULT 'member',
                    -- 'owner', 'admin', 'member'
    joined_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    added_by        BIGINT,
    is_muted        BOOLEAN NOT NULL DEFAULT FALSE,
    left_at         TIMESTAMPTZ,  -- NULL = 在群中
    
    PRIMARY KEY (group_id, user_id)
);

CREATE TABLE group_events (
    id              BIGSERIAL PRIMARY KEY,
    group_id        VARCHAR(64) NOT NULL,
    event_type      VARCHAR(30) NOT NULL,
                    -- 'created', 'member_joined', 'member_left',
                    -- 'member_removed', 'name_changed', 'avatar_changed',
                    -- 'settings_updated'
    actor_user_id   BIGINT,
    target_user_id  BIGINT,
    payload         JSONB,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### 3.2 权限矩阵

```
                     owner    admin    member

添加成员               ✓        ✓

置顶群 (个人设置)      ✓        ✓        ✓
静音 (个人设置)        ✓        ✓        ✓
```

### 3.3 Conversation 与 Group 的关系

```
每个群对应一个 Conversation:
  - conversation.type = 'group'
  - conversation.group_id = groups.id
  - 成员列表 = group_members WHERE group_id = $gid AND left_at IS NULL

群创建时的原子操作:
  BEGIN
    INSERT INTO groups (...);
    INSERT INTO conversation (id, type, group_id, ...);
    INSERT INTO conversation_members (conversation_id, user_id, role)
      VALUES ($cid, $creator_id, 'owner');
    INSERT INTO outbox_events (..., 'group_created', ...);
  COMMIT;
```

## 4. 扇出策略

### 4.1 分级扇出

```
小群 (≤ 50 人):
  策略: 全写扩散
  实现:
    1. 消息写入 messages 表
    2. 为每个在线成员创建投递任务
    3. 为每个成员写入 sync_event (用于离线同步)
    4. 更新每个成员的 conversation_summary

中群 (51–200 人):
  策略: 在线实时投递 + 离线延迟同步
  实现:
    1. 消息写入 messages 表
    2. 在线成员 (查询路由表) → 实时投递
    3. 离线成员 → 仅写入 sync_events, 不在此时更新 summary
    4. 离线成员上线时按需计算 summary

大群 (> 200 人, P1):
  策略: 纯读扩散
  实现:
    1. 消息仅写入群信箱 (单条记录)
    2. 成员拉取时从群信箱读取
    3. 需要游标追踪每个成员在群信箱中的阅读位置

> 群成员上限 `max_members` 默认 200，与 P0 写扩散能力匹配。超过 200 人的大群作为 P1 学习扩展，引入读扩散模型。
```

### 4.2 扇出消费者（群消息路径）

```
群消息 Outbox 消费:

func processGroupMessage(event):
    group = getGroup(event.conversation_id)
    members = getActiveMembers(group.id)  // left_at IS NULL
    
    onlineDevices = []
    offlineUsers = []
    
    for member in members:
        if member.id == sender.id:
            continue  // 不投递给自己
        devices = lookupUserRoute(member.id)
        if devices is not empty:
            onlineDevices.append((member, devices))
        else:
            offlineUsers.append(member)
    
    // 在线成员: 实时投递 (并行，最多 10 个并发)
    parallel(limit=10, onlineDevices) { (member, devices) ->
        for device in devices:
            deliverToDevice(device.gateway_node, device.connection_id, message_frame)
    }
    
    // 全部成员: 写入同步事件 (批量 INSERT)
    batchInsertSyncEvents(members.filter(!= sender), message_event)
    
    // 更新 ConversationSummary (批量)
    batchUpdateConversationSummaries(members.filter(!= sender), group.id, message_preview)
```

#### 写扩散放大估算

以 200 人群、每秒 100 条消息为例：

| 操作 | 单次放大倍数 | 每秒写入量 |
|------|-------------|-----------|
| `messages` 主表 | 1 | 100 |
| `outbox_events` | 1 | 100 |
| `sync_events`（每个成员一条） | 199 | 19,900 |
| `conversation_summaries`（每个成员一行更新） | 199 | 19,900 |
| 在线投递 RPC | 取决于在线设备数 | ~5,000–20,000 |

**结论**：写扩散把“1 条群消息”放大为“N 次写入 + N 次投递”。这正是 P0 限制群规模、P1 引入读扩散的根本原因。

### 4.3 热点群保护

```
热点群判定 (1 分钟内消息 > 50 条):
  → 打标记: hot_group:{group_id} (Redis, TTL 60s)
  → 触发保护措施

保护措施:
  1. 限流: 该群消息写入限流到 50 条/秒 (超过返回 429)
  2. 投递降级: 在线投递合并为批次，每 500ms 批量投递一次（而不是逐条）
  3. 同步延迟: sync_events 写入延迟到秒级批量
  4. 摘要更新去抖动: 每 2 秒更新一次 conversation_summary
  5. 独立消费者: 热点群的 Outbox 事件由独立线程池处理
  6. 熔断: 如果扇出消费者延迟超过 10 秒 → 拒绝新消息 → 返回 "群聊繁忙"
```

### 4.4 扇出延迟 vs 吞吐权衡

```
正常情况下 (非热点):
  扇出消费者延迟: P95 < 200ms
  成员消息到达延迟: P95 < 500ms

热点群:
  扇出消费者延迟: P95 < 2s (批次合并后)
  成员消息到达延迟: P95 < 3s
  吞吐优先于延迟
```

## 5. 成员变更事件

### 5.1 加入群

```
流程:
  1. 添加者调用 POST /groups/{gid}/members {user_ids: [...]}
  2. 校验: 添加者有 admin 权限, 群未满
  3. 事务:
     INSERT INTO group_members → 如果曾退群, 清除 left_at
     INSERT INTO group_events (member_joined)
     INSERT INTO conversation_members
  4. Outbox: group_member_added 事件
  5. 为新成员生成历史消息的 sync_events (最近 50 条)
  6. 通知群内其他成员: "XXX 加入了群聊"
```

### 5.2 退出/被踢

```
流程:
  1. 事务: 
     UPDATE group_members SET left_at = NOW()
     UPDATE conversation_members SET is_active = false
     INSERT INTO group_events (member_left / member_removed)
  2. 清理: 被移出者的 conversation_summary 标记 is_hidden = true
  3. 通知群内其他成员
  4. 被移出者不能再向该群发消息 (403)
```

## 6. 群消息与 E2EE（P1 / 学习扩展）

WhatsApp 群聊使用 **Sender Key**（不是为每个成员单独加密）：

```
Sender Key 流程:
  1. 群创建时，每个成员生成一个 Sender Key (AES-256-CBC) 
     并用自己与每个其他成员之间的 pairwise 通道分发此 Sender Key
  2. 发送群消息时，用 Sender Key 加密一次
  3. 所有成员用同一 Sender Key 解密
  
优势: 加密一次，N 人解密，而非 N 次加密 N 次解密
限制: 成员变更时需要重新分发 Sender Key (ratchet forward)

详见 Spec 10。
```
