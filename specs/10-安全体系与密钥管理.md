# Spec 10 - 安全体系与密钥管理

## 1. 目标

定义聊天系统的安全边界、基础防护策略和密钥管理扩展点，使系统在认证、传输、存储和设备层具备可演进的安全架构。

- **P0**：建立可验证的最小安全基线，保证学习系统在公开网络中安全运行。
- **P1 / 学习扩展**：逐步引入设备公钥体系、端到端加密（E2EE）和群密钥更新策略。

> 安全设计不是额外附加层，而是贯穿登录、连接、消息、媒体和同步的横切能力。P0 不追求生产级 E2EE，但必须消除常见高危漏洞。

## 2. 威胁模型

### 2.1 攻击者假设

| 能力 | 描述 | 本阶段防护目标 |
|------|------|---------------|
| 网络监听者 | 可截获公网流量 | 全链路 TLS，防止明文泄露 |
| 凭证窃取者 | 可获取用户密码/Token | 短周期 Token + Refresh Rotation（P0 可选） |
| 内部人员 | 可接触数据库/日志 | 敏感字段加密、日志脱敏 |
| 恶意上传者 | 上传非法文件 | 基础文件类型/大小校验（P0），内容审核（P1） |
| 重放攻击者 | 重放历史请求 | 幂等键、Token jti、请求签名 |

### 2.2 不在 P0 威胁模型内

- 服务端被完全攻破后的消息内容保护（需 E2EE）。
- 国家级流量分析或中间人攻击（需证书固定、E2EE）。
- 客户端越狱/Root 后的密钥提取。

```
       │                                │                            │
```

## 4. 最小安全基线（P0）

### 4.1 传输安全

- 所有客户端-服务端通信强制 HTTPS / WSS。
- 禁用 TLS 1.0/1.1，最低 TLS 1.2，推荐 TLS 1.3。
- HSTS 头配置。
- 证书由受信任 CA 签发，P1 可考虑证书固定（Certificate Pinning）。

### 4.2 认证与授权

- Access Token 短期有效（1 小时），JWT 使用 RS256 签名。
- Refresh Token 使用 opaque 随机串，服务端存储哈希。
- 关键操作（如吊销设备、删除账号）需重新验证身份。
- 设备会话与 `device_id` 绑定，Token 泄露后可单独吊销设备。

### 4.3 存储安全

| 数据 | 存储方式 | 说明 |
|------|----------|------|
| 用户手机号 | 数据库明文存储，访问需审计 | P0 学习系统；P1 可考虑哈希 |
| Refresh Token | 服务端存储 SHA-256 哈希 | 原始值不落地 |
| 推送令牌 | 数据库存储，日志脱敏 | 避免泄露用户设备标识 |
| IP 地址 | 审计表加密存储 | 满足审计与隐私平衡 |
| 消息内容 | P0 服务端明文存储 | P1 E2EE 后服务端不持有明文 |

### 4.4 上传下载安全

- 媒体上传使用 presigned URL，限制 MIME 类型、文件大小、过期时间。
- 下载 URL 带 HMAC 签名，包含 `object_key + user_id + expires_at`。
- 下载前校验用户是否属于对应会话。

### 4.5 日志与审计

- 不记录消息内容到系统日志。
- 不记录完整 Token 到日志。
- 记录关键安全事件：登录成功/失败、设备增删、Token 刷新/吊销、异常访问。

## 5. 最小安全基线 Checklist（P0）

```
□ 全链路 HTTPS / WSS
□ TLS 1.2+，禁用弱密码套件
□ JWT 短期有效 + 签名验证
□ Refresh Token 哈希存储
□ 敏感字段不落日志
□ 上传文件 MIME/大小校验
□ 下载 URL 限时签名
□ 关键操作审计
□ 设备会话可吊销
□ 接口限流与防暴力破解
```

## 6. 密钥管理预留设计（P1 / 学习扩展）

### 6.1 设备密钥体系

```
Identity Key Pair (IK):
  - 每设备生成 Ed25519/X25519 密钥对
  - 公钥注册到服务端，用于身份认证与 E2EE
  - 私钥仅存储于客户端 Keychain / Keystore

Signed PreKey (SPK):
  - 中等周期轮换（如 1–3 个月）
  - 用于建立初始会话密钥

One-Time PreKey (OPK):
  - 一次性使用，用完即删
  - 用于防止未来密钥泄露被回溯解密
```

### 6.2 服务端密钥管理

```
密钥存储:
  - 生产环境使用 KMS/HSM（如 AWS KMS、HashiCorp Vault）
  - JWT 签名私钥不随代码部署
  - 数据库加密密钥与业务逻辑分离

密钥轮换:
  - JWT 签名密钥定期轮换（如每季度）
  - 旧密钥保留一个 TTL 窗口，保证已签发 Token 仍能验证
  - 轮换过程对客户端透明
```

## 7. 端到端加密（E2EE）路线图（P1）

### 7.1 私聊 E2EE

- 参考 Signal Protocol / Double Ratchet。
- 每条消息使用独立消息密钥。
- 服务端只转发密文，无法解密。

### 7.2 群聊 E2EE（Sender Key）

- 详见 Spec 07 §6。
- 群成员变更时重新分发 Sender Key 并 ratchet forward。

### 7.3 E2EE 对系统的影响

- 服务端无法做内容搜索、内容审核、关键词过滤。
- 推送通知无法直接展示明文，需 Notification Service Extension（Spec 09 §3.3）。
- 多端同步需同步密钥状态，复杂度显著增加。

## 9. 流程描述

P0 阶段重点建立可验证的最小安全基线：全链路加密、短周期 Token、敏感字段保护、审计日志。P1 阶段在此基础上引入设备公钥体系与 E2EE，逐步提升系统到更高隐私等级。所有安全能力都应通过代码评审、渗透测试和配置检查来验证，而不是仅依赖文档声明。
