# Spec 01 - 产品边界与 SLO

## 1. 目标

定义系统第一阶段要解决的问题、非目标范围以及核心服务等级目标，防止产品与技术范围失控。

> **项目定位**：本系统是一个用于**学习大规模高负载实时聊天处理技术**的学习/实验系统。文中的 SLO、容量假设与 SLI 主要用于**学习评估、压测参照与架构取舍训练**，不是面向用户的商业 SLA。

## 2. 业务目标

- 提供稳定的私聊与群聊能力。
- 支持离线恢复和多端同步。
- 支持图片等基础媒体消息。
- 支持基础送达与已读反馈。

## 3. 非目标（P1 / 学习扩展）

- 不在第一阶段实现完整音视频通话。
- 不在第一阶段实现支付与开放平台。
- 不在第一阶段实现生产级全量 E2EE。
- 不在第一阶段实现联系人发现的哈希隐私匹配。
- 不在第一阶段实现媒体内容审核、视频转码与 CDN 全球分发。
- 不在第一阶段实现 Notification Service Extension 解密推送预览。

> **原则**：任何非 P0 的能力都不应成为 P0 主链路的隐性依赖。P0 设计应保留扩展点，但核心链路必须能在没有 P1 能力的情况下独立运行。

## 4. 核心挑战

- 产品范围过大导致架构提前复杂化。
- 未定义性能目标，后续无法判断是否达标。
- 没有容量基线，系统扩展策略失真。

## 5. 典型方案

- 采用分阶段能力集，先固化文本消息主链路。
- 把体验指标转化为可监控的 SLI/SLO。
- 为峰值流量、连接数、会话规模定义初始容量假设。

## 6. 建议 SLO

### 6.1 可用性

- 消息接收 API 可用性 >= 99.9%
- 长连接接入成功率 >= 99.5%
- 增量同步成功率 >= 99.9%

### 6.2 时延

- 文本消息接收确认 P95 < 300ms
- 在线消息投递 P95 < 800ms
- 离线同步首屏恢复 P95 < 2s

### 6.3 正确性

- 消息重复展示率 < 0.01%
- 会话内顺序错误率 < 0.01%
- 多端已读状态不一致率 < 0.1%

## 7. 容量假设（学习型规模）

以下数字用于**理解高负载场景下的架构取舍**，不承诺真实生产水位。

| 指标 | 假设值 | 学习目的 |
|------|--------|----------|
| DAU | 10 万 – 100 万 | 理解连接池、分片与缓存设计 |
| 峰值在线连接 | 5 万 – 20 万 | 理解长连接网关与 C10K→C1000K 问题 |
| 峰值消息写入 | 1 千 – 1 万条/秒 | 理解写扩散放大与 Outbox 消费 |
| 单个热点群峰值 | 每秒数百到数千消息 | 理解热点行、限流与降级策略 |

## 8. 流程描述

产品规划阶段先定义 P0 功能边界，再把功能映射为链路与数据目标，最后为每条关键链路定义量化指标。没有 SLO 的功能只代表“存在”，不代表“可交付”。
