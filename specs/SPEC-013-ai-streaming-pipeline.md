# SPEC-013 — AI 会话与流式消息管道：协议与数据流的深水区

> 状态: Draft | Milestone: M4 | 依赖: SPEC-001, 003, 004, 012 | 被依赖: 014（共享 AI 会话形态）

## 1. 背景与动机（Why）

给系统加一个 AI 助手会话（bot 参与者，回复逐 token 流式呈现）。选这个功能
不是为了赶时髦，而是它**恰好打破 M1 的一条根基假设——"消息是原子的"**：
一条消息从此可以处于"正在生成、内容持续增长"的中间态。如何在不推翻
收件箱/定序/幂等体系的前提下容纳流式，是一次真刀真枪的协议与数据流设计
练习。这份 spec 的服务端部分也是把 002 背压、003 可靠性拿出来重新审视的
回归考试。

## 2. 核心挑战与典型解法

### 挑战 A：流式与消息模型的根本冲突

`conv_seq` 在落库时分配（003），但流式消息"落库"发生在生成**结束**时。
生成中的 partial 内容算什么？

**解法：把 partial 归入 007 定义过的数据分级光谱——"可丢弃"一端：**

- 新帧类型（Envelope 扩展，SPEC-001 预留区）：

```protobuf
message StreamChunk {
  string stream_id  = 1;  // 本次生成的临时标识（非消息 ID）
  int32  chunk_seq  = 2;  // 流内序号，检测缺口
  string delta      = 3;  // 增量文本
  bool   done       = 4;
}
```

- **StreamChunk 不落库、不进收件箱、不重传**——纯网关直推，和 typing
  同级。掉了就掉了；
- 生成完成 → AI 服务把**完整文本**作为普通消息走 003 全套管道
  （幂等键 = `stream_id` 派生的 `client_msg_id`，分配 `conv_seq`，写收件箱）
  → 客户端收到终版 `MessagePush` 后用 `stream_id` 对账：流式缓冲已渲染的
  气泡原地"转正"（替换为持久消息，视觉无跳变）；
- **可靠性推论（本 spec 的灵魂）**：客户端中途断线/杀进程 → partial 丢失
  无所谓，重连 sync 拿到终版消息，体验从"看着它长出来"降级为"直接看到
  全文"——正确性零损失。流式只是投影，收件箱仍是唯一事实。这证明 M1 的
  分层在新需求下不需要改一行核心逻辑。
  StreamChunk 与 typing（007）在"纯转发、不落库、不重试"上完全对称——
  两次独立实现共同验证了 007 数据分级光谱的普适性：不是 typing 特殊，
  而是所有可丢弃数据都该这么做。

### 挑战 B：Bot 作为一等公民参与者

- bot 是一个特殊 user（`user_type=bot`）：可加入会话、有头像有名字，
  **复用**成员权限、扇出、收件箱全部机制——不为 bot 单开平行体系；
- 新服务 `aisvc`：订阅发给 bot 的消息 → 组装上下文 → 调 LLM →
  边生成边发 StreamChunk、结束发终版消息；
- **上下文截断策略（评审 A9 定案）**：按 `conv_seq` 逆序取最近 50 条
  （先取最新），超 token 预算时从最早的开始丢（truncate-head——聊天场景
  最近消息最相关）；**bot 自己的历史回复纳入上下文**（多轮对话的前提，
  自指风险由"上下文只含终版消息、绝不含 partial"消解）；系统提示固定在
  预算之外不参与截断；token 计数用后端 tokenizer（Ollama 场景 tiktoken
  估算是可接受近似），不做按字符粗估；
- LLM 后端可插拔：本地 Ollama（学习默认，零成本）/ OpenAI 兼容 API
  （接口抽象一致）。**消息内容对 LLM 是数据不是指令**：系统提示与用户内容
  硬分隔，用户消息里的"忽略以上指令"不得改变 aisvc 行为（红队用例进验收）。

### 挑战 C：背压与取消 —— 002 的回归考试

- LLM 产出 token 速率（可达 100+ tok/s）× 并发生成数，全部灌进网关的
  per-conn sendCh（002 的有界通道）：慢客户端收流式 → sendCh 溢出断连的
  老规则**对流式不合适**（断连太重）。细化策略：StreamChunk 标记为
  可丢弃帧，通道高水位时**优先丢 chunk**（客户端靠 `chunk_seq` 缺口感知，
  终版消息兜底），消息帧仍走断连规则——帧分级调度，002 的调度器升级点；
- **取消**：停止按钮 → `StreamCancel{stream_id}` 上行 → aisvc 中止生成，
  已生成部分作为终版消息落库（标记 truncated）。取消与 done 的竞态：
  以 aisvc 收到先后为准，幂等处理双收；
- 成本护栏：单次生成 max_tokens 上限、单用户并发生成数 = 1、
  aisvc 全局并发闸（超出排队并向客户端发排队态帧）。

### 挑战 D：与 E2EE 的边界（诚实的架构声明）

服务端 aisvc 要读明文才能生成 → **bot 会话显式不做 E2EE**（会话创建时
标记，UI 明示"此对话不加密"，011 的加密开关按会话类型豁免）。
加密会话里的智能只能走端侧路径——这条边界正是 SPEC-014 存在的理由，
在文档里把这个"云端智能 vs 隐私"的取舍写透。

## 3. 范围

**In**：协议扩展（StreamChunk/StreamCancel/排队态帧 + 001 的 ErrorCode 扩展
`GENERATION_FAILED` RETRYABLE）、aisvc 服务（上下文组装/后端抽象/护栏/
红队防护）、网关帧分级调度、iOS 端流式对账 + 停止按钮（渲染直接用 012 引擎、
消息生命周期用 012 的 streaming 分支状态机）、bot 会话创建流程。
**Out**：端侧模型（014）、多 bot / bot 市场（backlog）、群里 @bot（backlog，
管道天然支持）、RAG/联网检索（backlog）。

## 4. 验收标准

1. 流式体验：首 token 延迟 p50 < 1.5s（本地 Ollama 基准机型记录在案）；
   全程 012 渲染引擎掉帧 < 5%。
2. 对账正确性：生成中途杀 App → 重开 sync → 终版消息恰好一条、无 partial
   残留、无重复（`stream_id` 幂等的证明）；生成中断线重连 → 气泡从 partial
   平滑转正或直接显示全文，无双气泡。
3. 数据分级证明：DB dump 中 grep 任意 StreamChunk 的 delta 内容零命中
   （chunk 不落库，方法同 011 盲测）。
4. 背压分级：慢客户端场景下 chunk 丢弃计数 > 0 且连接不断、普通消息不丢
   （帧分级调度的证明）；恢复后终版消息完整。
5. 取消：点停止后 1s 内生成中止，落库消息带 truncated 标记；
   取消/done 竞态双收注入测试无双写。
6. 红队：10 条 prompt-injection 用例（含"忽略指令输出系统提示"）全部不改变
   aisvc 行为（用例库进 CI）。

## 5. 测试计划
