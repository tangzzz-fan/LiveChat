# SPEC-010 — 多设备同步：从"用户"到"设备"的维度升级

> 状态: Draft | Milestone: M3 | 依赖: SPEC-003 | 被依赖: 011（E2EE 会话是设备对级别）

## 1. 背景与动机（Why）

M1 里"接收者"= user_id，一个隐含假设是一人一设备。多设备（iPhone + iPad）
把系统里所有以 user 为轴的状态拆成两层：**有些状态属于用户（消息内容、群成员
身份），有些属于设备（投递水位、连接、推送 token、加密会话）。** 分不清这两层
是多设备 bug 的总根源。这也是 E2EE（011）的硬前置：Signal 会话建立在设备对上。

## 2. 核心挑战与典型解法

### 挑战 A：投递游标从 per-user 变 per-device

iPhone 已收到消息 ≠ iPad 也收到了。**解法：**
- `inbox` 保持 per-user 不变（写扩散成本不随设备数放大——这是关键设计，
  收件箱是用户级的"事实"）；
- 游标变成 per-device：`sync_state(user_id, device_id, last_ack_inbox_seq)`。
  每台设备独立执行 SPEC-003 的游标同步协议，互不干扰；
- **最慢设备不拖累回收（评审 C1）**：inbox 条目的删除**不等待**全部设备
  ACK，遵循 SPEC-003 的保留策略（`max(30d, 全设备 max(last_active_at)+7d)`
  后物理删除）。长期不上线的设备重新连上时游标已出窗 → 服务端返回
  `gap_detected=true` → 该设备降级为 per-conversation `conv_seq` 范围拉取。
  两份 spec 引用同一条策略，语义以 SPEC-003 为准；
- 路由表升级：`route:{uid}` 从单值变集合 `{device_id → gateway_id}`，
  在线推送遍历该用户所有在线设备。
  **key 格式迁移（评审 A2）**：不改旧 key 的 value 格式，改用新 pattern
  `route:{uid}:{device_id}`（在线推送用 `SCAN`/预存的 device 列表拼 key 批量
  MGET）。上线次序：先双读（新 key 优先，miss 时 fallback 旧 `route:{uid}`），
  写只写新 key；旧 key 靠原有 TTL（90s）自然过期，双读代码在一个版本周期后
  删除。零停机、无离线迁移窗口。

### 挑战 B：自己发的消息要出现在自己的其他设备上

iPhone 发出的消息，iPad 也得显示（WhatsApp 早期最著名的缺陷之一，
Web 版必须手机在线就是这么来的）。**解法：回环投递**——写扩散时发送者
本人也进 inbox（M1 里其实就该这么做，M1 单设备时它只是"跳过自己当前连接"
的推送优化；多设备下发送者的**其他**设备照常推送 + 游标补拉）。
客户端按 `client_msg_id` 识别"这是我自己发的"做去重合并（本地 outbox 里
有它 → 升级状态而非插入新行）。

### 挑战 C：设备注册、信任与撤销

- `devices(user_id, device_id, name, platform, created_at, last_active_at,
  revoked_at)`；登录 = 注册设备，token/加密身份都挂在 device 下；
- 设备列表管理 UI（"已登录的设备"）+ 远程登出：撤销 = 踢连接 + 删推送 token
  + 拒绝该 device 的后续鉴权 + （011 后）其他人停止向它加密；
- 设备数上限 5（WhatsApp companion 上限相同——控制 E2EE 的会话扇出规模，
  又一个产品约束源自架构约束的例子）。

### 挑战 D：新设备的历史消息

新登录的 iPad 怎么拿到历史聊天记录？三条路线，取舍是本 spec 的核心讨论：

| 路线 | 机制 | 采用者 |
|------|------|--------|
| A. 服务端拉取 | 新设备从 messages 表按会话分页拉 | Telegram（云聊天，非 E2EE） |
| B. 老设备转移 | 主设备打包加密历史 → 经服务器中转给新设备 | WhatsApp 多设备 |
| C. 不同步 | 新设备只看登录后的消息 | Signal（iPad 配对早期行为） |

**决定：M3 先实现 A**（明文阶段最直接，学分页回填管道：按会话优先级
——最近活跃优先——渐进回填，不阻塞新消息）；**文档阐述 011 上线后 A 失效**
（服务端只有密文且无密钥）**必须切 B**，并设计 B 的传输格式预留。
这个"加密如何改变架构"的对照本身就是学习目标。

## 3. 范围

**In**：devices 表与注册/撤销链路、per-device 游标、路由表集合化、回环投递
与客户端合并、设备管理 UI、新设备历史回填（路线 A）、APNs token 挂靠 device
（008 的字段本来就有，这里接通语义）。
**Out**：E2EE 设备身份密钥（011）、跨设备 draft/阅读位置同步（backlog）、
桌面/Web 端（D8 排除——用第二台 iOS 设备/模拟器验证）。

## 4. 验收标准

1. 双设备在线：iPhone 发消息，iPad < 1s 内出现且状态正确（显示为"我发的"，
   非对方消息；抓帧确认走的是推送而非轮询）。
2. 游标独立：iPad 关机一天，iPhone 正常收发；iPad 开机后独立补拉齐平，
   iPhone 的水位全程不受影响（DB 断言）。
3. 新设备回填：老账号（1w 条历史、20 个会话）登录新模拟器 → 最近 5 个会话
   优先可见 < 10s，全量回填后台完成，期间新消息实时到达不被阻塞。
4. 远程登出：iPhone 上撤销 iPad → iPad 连接 3s 内被踢、重连被拒、
   不再收到任何推送（mock APNs 证明）。
5. 回环去重：发送方设备自己的 sync 结果里包含自己发的消息 → 本地不出现
   重复行（单测 + UI 断言）。

## 5. 测试计划

per-device 游标与回环合并单测；双模拟器 XCUITest 冒烟（发/收/撤销）；
loadtest 增加 multi_device.yaml（每用户 2~3 虚拟设备，验证路由集合与
写扩散成本不随设备数放大——inbox 写入量指标为证）。
