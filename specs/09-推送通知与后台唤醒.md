# Spec 09 — 推送通知与后台唤醒

## 1. 目标

建立 iOS 端离线、后台或长连接不可达时的系统级推送唤醒机制。推送是消息同步的**触发器**，不是消息本身的**载体**。

- **P0**：Silent Push 后台唤醒、Visible Push 通知栏提醒、推送去重、频控节流、后台同步、Badge 更新。
- **P1 / 学习扩展**：Notification Service Extension 解密推送预览（用于 E2EE 场景）。

## 2. 核心挑战

### 2.1 推送的不确定性

APNs 是 Best-Effort 的——不保证送达、不保证顺序、不保证不重复。Apple 明确文档："Delivery of notifications is not guaranteed."

这意味着：推送可以做唤醒触发器，不能做消息真相。

### 2.2 在线投递与推送的重复

设备在 WebSocket 断开后的 100ms 内，服务端可能同时触发了推送和重连。如果处理不当：
- 同一消息在 WebSocket 和推送中各出现一次
- 用户看到重复通知

### 2.3 E2EE 下的隐私推送（P1 / 学习扩展）

WhatsApp 的核心体验：通知栏显示的消息预览是**客户端解密后的明文**。这需要：
- 服务端无法解密消息内容
- 但服务端需要生成推送通知
- 解：服务端推送「有消息，请来同步」，客户端 Notification Extension 解密后修改通知内容

> P0 阶段：Visible Push 使用固定隐私文案（如 "🔒 新消息"），不实现 Extension 解密。

### 2.4 iOS 后台限制

iOS 对后台任务有严格限制：
- 普通 App：后台 30 秒执行窗口
- VoIP App：有特殊推送通道但必须是真正的 VoIP 应用
- BGTaskScheduler：系统决定执行时机，不可靠
- Notification Service Extension：30 秒窗口，P1 用于解密消息预览

## 3. 推送架构

### 3.1 决策树：是否推送

```
消息到达 → 消息扇出完成 → 每个目标设备:
  │
  ├─ 设备在线 (WebSocket connected)
  │   └─ 直接 WebSocket 投递 → 不推送
  │
  ├─ 设备离线 / 后台 且 WebSocket disconnected
  │   ├─ App 在前台 → 不推送 (用户能看到)
  │   ├─ App 在后台 (Background)
  │   │   └─ 发送 Silent Push (content-available: 1)
  │   │      → 唤醒 App → 触发后台同步 → 下载消息
  │   │
  │   └─ App 被杀 (Killed / Suspended)
  │       └─ 发送 Visible Push (alert + sound)
  │          → 用户点击 → App 启动 → 全量同步
  │
  └─ 推送令牌过期 / 不可用 → 仅写 sync_events → 等下次上线
```

### 3.2 推送内容策略

```
Silent Push (后台唤醒):
{
  "aps": {
    "content-available": 1,         // 静默推送
    "apns-priority": 5             // 低优先级, 节能
  },
  "sync_trigger": {
    "latest_event_seq": 15200,     // 最新事件序号
    "reason": "new_message"        // message / group_event / receipt
  }
}

Visible Push (通知栏, P0):
{
  "aps": {
    "alert": {
      "title": "张三",
      "body": "🔒 新消息"          // 默认隐私文案
    },
    "sound": "default",
    "badge": 3,
    "apns-push-type": "alert"
  },
  "sync_trigger": {
    "latest_event_seq": 15200
  }
}

Visible Push (通知栏, P1 E2EE 扩展):
{
  "aps": {
    "alert": {
      "title": "张三",
      "body": "🔒 新消息"
    },
    "sound": "default",
    "badge": 3,
    "mutable-content": 1,          // 触发 Notification Service Extension
    "apns-push-type": "alert"
  },
  "sync_trigger": {
    "latest_event_seq": 15200
  },
  "encrypted_preview": "<base64_encrypted_preview>"  // E2EE 加密预览
}
```

### 3.3 Notification Service Extension（P1 / 学习扩展：E2EE 通知解密）

```
// NotificationService.swift
class NotificationService: UNNotificationServiceExtension {
    override func didReceive(_ request: UNNotificationRequest,
                             withContentHandler contentHandler: @escaping (UNNotificationContent) -> Void) {
        
        let bestAttempt = request.content.mutableCopy() as! UNMutableNotificationContent
        
        // 1. 检查是否有加密预览
        if let encryptedPreview = request.content.userInfo["encrypted_preview"] as? String {
            
            // 2. 用本地密钥解密
            if let decrypted = E2EEManager.shared.decryptNotificationPreview(
                encryptedPreview: encryptedPreview,
                senderId: senderId
            ) {
                bestAttempt.body = decrypted  // "今晚一起吃饭？"
            } else {
                bestAttempt.body = "🔒 新消息"  // 解密失败，保持隐私
            }
        }
        
        // 3. 更新角标
        if let badge = request.content.userInfo["badge"] as? Int {
            bestAttempt.badge = NSNumber(value: badge)
        }
        
        contentHandler(bestAttempt)
    }
    
    override func serviceExtensionTimeWillExpire() {
        // 30 秒超时 → 使用隐私文案
        bestAttempt.body = "🔒 新消息"
        contentHandler(bestAttempt)
    }
}
```

### 3.4 推送去重

```
客户端去重策略:

收到推送时:
  1. 提取推送中的 conversation_id / server_message_id (如果有)
  2. 检查本地 DB:
     - UNIQUE(server_message_id) 已存在 → 忽略, 不更新 UI
     - 不存在 → 触发同步
  3. WebSocket 投递到达时不再弹系统通知 (仅在 App 内展示)

时间窗口去重:
  - 维护一个最近 60 秒内处理的推送 ID 集合 (内存 LRU)
  - 新推送到达时检查去重表
```

### 3.5 频控与节流

```
推送频率限制 (服务端):
  - 同一会话: 30 秒内最多 1 条 visible push
    (如果 30 秒内有多条消息, 合并为 "张三: 3 条新消息")
  - 同一用户: 每秒最多 1 条推送
  - 群聊: 如果用户已静音 → 不发 visible push

Badge 更新:
  - badge = 所有未读会话的未读消息总数
  - 每次推送时更新 badge
  - 用户打开 App 后 badge 清零
```

## 4. iOS 后台唤醒流程

### 4.1 Silent Push → 后台同步

```
Silent Push 到达
  │
  ├─ iOS 判断: App 是否在后台?
  │   ├─ 是 → 分配 30 秒后台执行时间
  │   └─ 否 → App 在前台, 忽略 Silent Push
  │
  ├─ AppDelegate.didReceiveRemoteNotification(fetchCompletionHandler:)
  │   │
  │   ├─ 1. 发起 WebSocket 连接 (或 HTTP 同步)
  │   ├─ 2. 增量同步: GET /sync/events?cursor=local_cursor
  │   ├─ 3. 将新消息存入本地 DB
  │   ├─ 4. 若消息与当前前台会话相关 → 本地通知 (Local Notification)
  │   │    若无关 → 不弹通知 (避免重复系统推送)
  │   └─ 5. fetchCompletionHandler(.newData)
  │
  └─ 超时 (30 秒)
      └─ fetchCompletionHandler(.failed)
```

### 4.2 BGTaskScheduler（定期后台刷新）

```
// 注册后台任务
BGTaskScheduler.shared.register(forTaskWithIdentifier: "com.livechat.sync",
                                using: nil) { task in
    self.handleBackgroundSync(task: task as! BGAppRefreshTask)
}

func handleBackgroundSync(task: BGAppRefreshTask) {
    // 1. 设置过期处理
    task.expirationHandler = {
        // 任务即将被系统杀死, 快速收尾
        self.syncTask?.cancel()
    }
    
    // 2. 增量同步
    let result = await SyncService.shared.syncIncremental()
    
    // 3. 调度下一次后台刷新
    let nextRequest = BGAppRefreshTaskRequest(identifier: "com.livechat.sync")
    nextRequest.earliestBeginDate = Date(timeIntervalSinceNow: 15 * 60) // 最少 15 分钟
    try? BGTaskScheduler.shared.submit(nextRequest)
    
    // 4. 完成
    task.setTaskCompleted(success: result.isSuccess)
}
```

**注意：** BGTaskScheduler 的执行时机由系统决定，不可靠。它更像是「顺便刷新」而非「及时通知」。真正可靠的消息到达仍需通过 APNs Push。

### 4.3 App 启动后的恢复流程

```
App 从推送启动 / 用户手动启动:
  
  1. AppDelegate.didFinishLaunching
     → 如果从推送启动: 
        检查 launchOptions[.remoteNotification]
        直接跳转到对应会话
        
  2. 恢复本地数据库 (如果使用 CoreData/GRDB)
     → 检查上次同步游标
     
  3. 建立 WebSocket 连接
     → 握手时携带 last_event_seq
     
  4. 触发增量同步
     → 拉取离线期间的所有事件
     
  5. 如果 last_event_seq 落后过多 (> 30 天):
     → 全量重建会话列表
```

## 5. 推送服务端架构

```
Push Orchestrator (推送编排服务):

1. 从 Outbox Consumer 接收推送任务
2. 去重检查 (同一用户+会话, 30s 内已有推送 → 合并)
3. 查询推送令牌:
   SELECT push_token, platform
   FROM device_sessions
   WHERE user_id = $uid AND push_token IS NOT NULL
4. 构建推送 payload (区分 iOS APNs / Android FCM)
5. 调用推送服务 (HTTP/2 → APNs)
6. 记录推送事件:
   INSERT INTO push_events (user_id, device_id, push_type, 
      conversation_id, apns_response, created_at)
7. 监控:
   - APNs 拒绝 (BadDeviceToken → 标记设备令牌失效)
   - APNs 延迟
   - 推送 → 用户启动 App 的转化率
```

## 6. 推送事件表

```sql
CREATE TABLE push_events (
    id              BIGSERIAL PRIMARY KEY,
    user_id         BIGINT NOT NULL,
    device_id       VARCHAR(64),
    push_type       VARCHAR(20) NOT NULL,  -- 'silent', 'visible', 'voip'
    conversation_id VARCHAR(64),
    message_id      VARCHAR(64),
    apns_response   JSONB,                -- APNs 返回的 response
    apns_status     VARCHAR(20),          -- 'sent', 'rejected', 'error'
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    
    INDEX idx_push_events_user_time (user_id, created_at DESC)
);
```
