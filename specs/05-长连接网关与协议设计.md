# Spec 05 — 长连接网关与协议设计

## 1. 目标

建立客户端与服务端之间的实时双向通信通道，定义 WebSocket 协议帧格式、opcode 枚举、心跳保活、用户路由和连接管理。网关不承载业务逻辑，只做协议转发和连接生命周期管理。本 spec 重点学习 C10K→C1000K 长连接管理、连接迁移与重连风暴防御。

## 2. 核心挑战

### 2.1 海量连接的资源成本

- 每个 WebSocket 连接 = 1 个 TCP socket = 1 个文件描述符 + 内核缓冲区
- Linux 默认 fd 限制 1024，需要调至百万级
- 每个连接的 goroutine/thread 开销不可忽略

### 2.2 移动端连接波动

- 前后台切换 → 系统可能断开 socket
- WiFi ↔ 蜂窝切换 → IP 变化 → 需要连接迁移
- 弱网 → 心跳超时 → 假死检测

### 2.3 重连风暴

网关重启或网络闪断后，数万设备几乎同时重连 → 瞬时连接洪峰 → 网关雪崩 → 更多断开 → 更多重连（正反馈）。

## 3. 协议帧设计

### 3.1 帧格式（Binary WebSocket, Protobuf 编码）

```
┌─────────────────────────────────────────────────────────┐
│ Byte 0    │ Byte 1-4   │ Byte 5-8    │ Byte 9-N        │
│ Version   │ Opcode     │ Payload Len │ Payload         │
│ (1 byte)  │ (4 bytes)  │ (4 bytes)   │ (variable)      │
└─────────────────────────────────────────────────────────┘

Version:  0x01 (协议版本)
Opcode:   大端 32-bit 无符号整数
Payload:  Protobuf 编码，具体结构由 opcode 决定
```

**Protobuf 定义：**

```protobuf
syntax = "proto3";

message WsFrame {
    uint32 version    = 1;  // 协议版本
    uint32 opcode     = 2;  // 操作码
    uint64 seq_id     = 3;  // 帧序号（客户端自增，用于 ACK）
    uint64 ack_seq_id = 4;  // 已确认的对方帧序号（捎带 ACK）
    bytes  payload    = 5;  // 业务负载
    
    // 元数据
    int64  timestamp_ms = 6;
    string trace_id     = 7;
}

// 握手（Client → Server）
message HandshakeRequest {
    string access_token  = 1;
    string device_id     = 2;
    string platform      = 3;  // "ios", "android", "desktop"
    string app_version   = 4;
    uint32 protocol_ver  = 5;  // 客户端支持的最高协议版本
    uint64 last_event_seq = 6; // 上次同步游标（断线重连时使用）
}

// 握手响应（Server → Client）
message HandshakeResponse {
    bool   success        = 1;
    string session_id     = 2;  // 本次连接的唯一 ID
    uint32 negotiated_ver = 3;  // 协商后的协议版本
    uint64 server_time_ms = 4;  // 服务端时间戳（用于时间同步）
    string error_message  = 5;
    uint32 heartbeat_interval_s = 6;  // 服务端建议的心跳间隔
    uint64 latest_event_seq = 7;      // 最新事件序号（用于判断是否需要同步）
}

// 心跳
message Heartbeat {
    // 空 payload 或可选携带客户端时间戳
}

message HeartbeatAck {
    uint64 server_time_ms = 1;
}

// ACK（业务级确认）
message MessageAck {
    uint64 event_seq    = 1;  // 确认的事件序号
    string ack_type     = 2;  // "delivered", "read"
    uint64 acked_at_ms  = 3;
}

// 错误
message ErrorFrame {
    uint32 error_code  = 1;
    string message     = 2;
    bool   should_reconnect = 3;
}
```

### 3.2 Opcode 枚举

```
OPCODE 范围规划:
  0x0000 - 0x0FFF : 系统协议 (握手、心跳、ACK、错误)
  0x1000 - 0x1FFF : 消息投递
  0x2000 - 0x2FFF : 状态同步
  0x3000 - 0x3FFF : 群组事件
  0x4000 - 0x4FFF : 预留(E2EE 密钥)
  0xF000 - 0xFFFF : 调试/内部

系统协议 (0x0000 - 0x0FFF):
  0x0001  HANDSHAKE_REQ      客户端 → 服务端  握手请求
  0x0002  HANDSHAKE_RESP     服务端 → 客户端  握手响应
  0x0003  HEARTBEAT          客户端 → 服务端  心跳
  0x0004  HEARTBEAT_ACK      服务端 → 客户端  心跳应答
  0x0005  ACK                双向            业务确认(delivered/read)
  0x0006  ERROR              服务端 → 客户端  错误通知
  0x0007  DISCONNECT         双向            通知对方即将断开
  0x0008  RECONNECT          客户端 → 服务端  快速重连(带 session_id)

消息投递 (0x1000 - 0x1FFF):
  0x1001  MESSAGE_DELIVERY   服务端 → 客户端  新消息投递
  0x1002  MESSAGE_STATUS     双向            消息状态变更通知

状态同步 (0x2000 - 0x2FFF):
  0x2001  SYNC_EVENT         服务端 → 客户端  增量同步事件
  0x2002  CONVERSATION_UPDATE 服务端 → 客户端  会话摘要变更
  0x2003  PRESENCE_CHANGE    双向            在线状态变更
  0x2004  TYPING_INDICATOR   双向            正在输入

群组事件 (0x3000 - 0x3FFF):
  0x3001  GROUP_CREATED
  0x3002  GROUP_MEMBER_ADDED
  0x3003  GROUP_MEMBER_REMOVED
  0x3004  GROUP_SETTINGS_UPDATED
```

## 4. 心跳与假死检测

### 4.1 心跳策略

```
客户端:
  - 每 30 秒发送 HEARTBEAT (服务端可在握手时调整此间隔)
  - 若 10 秒内未收到 HEARTBEAT_ACK → 认为连接假死 → 主动断开重连
  - 前后台切换时:
    前台 → 后台: 保持连接, 心跳间隔延长到 120s
    后台 → 前台: 恢复 30s 心跳, 检测是否需要补拉

服务端:
  - 若 90 秒内未收到任何帧 (包括 HEARTBEAT) → 主动断开
  - 断开前发送 DISCONNECT (code=timeout)
  - 清理该设备的在线路由信息

参数调优:
  心跳间隔 = 30s (常规)
  读超时   = 90s (3 次心跳窗口)
  写超时   = 30s
```

### 4.2 自适应心跳（弱网优化）

```
若连续 N 次心跳丢失:
  N = 1: 心跳间隔降至 15s（更频繁探测）
  N = 2: 心跳间隔降至 10s
  N = 3: 判定断开，启动重连

且应用 jitter:
  heartbeat_interval_actual = base_interval - random(0, 5)秒
  // 避免客户端同时心跳 → 服务端周期性压力尖峰
```

## 5. 用户路由表

### 5.1 路由模型

```
Redis 存储:

# 用户 → 网关节点
KEY:   gateway:user:{user_id}:{device_id}
VALUE: {gateway_node_id}:{connection_id}
TTL:   300s (与心跳关联, 心跳续期)

# 网关节点 → 连接列表(用于节点下线清理)
KEY:   gateway:node:{node_id}:connections
VALUE: SET {user_id}:{device_id}, ...

# 网关节点心跳
KEY:   gateway:node:{node_id}:heartbeat
VALUE: {timestamp}
TTL:   30s (网关每 10s 续期一次)
```

### 5.2 消息投递路由

```
FanoutService 查询路由:

1. 对每个目标成员:
   devices = SMEMBERS gateway:user:{user_id}:*
   
2. 对每个在线设备:
   gateway_node = GET gateway:user:{user_id}:{device_id}
   → 向对应网关节点发起 gRPC 调用: DeliverMessage(device, frame)
   
3. 设备不可达 (Redis 中无记录):
   → 走离线同步路径 (写入 sync_events 表)
```

## 6. 重连风暴防御

### 6.1 客户端退避

```
退避算法:
  attempt 0: 0.5s - 1s   (首次, 小间隔)
  attempt 1: 1s - 2s
  attempt 2: 2s - 4s
  attempt 3: 4s - 8s
  attempt 4: 8s - 16s
  attempt 5+: 16s - 30s (封顶)
  
  每次加随机抖动 ±25%

// 紧急情况: 如果连接已存活 > 5 分钟, 断开后:
// → 尝试快速重连 (RECONNECT opcode, 带旧 session_id)
// → 若失败, 回退到标准退避
```

### 6.2 服务端保护

```
1. 连接限流:
   - 每个 IP 每秒最多 5 个新连接
   - 每个 user_id 每秒最多 2 个新连接
   - 全局新连接速率超过网关容量 80% 时，对非重连请求返回 503 + Retry-After
   
2. 优先级队列:
   - 重连请求 (带 valid session_id, TTL 内未过期) → 高优先级
   - 新连接 → 正常优先级
   - 同一 user_id 的多个并发连接请求，只处理第一个，其余返回 429
   
3. 负载均衡:
   - 新连接路由到连接数最少的网关节点 (Least Connections)
   - 一致性哈希: 同一 user_id → 优先路由到历史节点
   
4. 熔断与降级:
   - 单网关节点 CPU > 85% 持续 10 秒 → 暂停接收新连接，由 LB 引流
   - 连接建立平均耗时 > 1s → 触发扩容或限流
```

```
└──────────────────┬─────────────────────────┘
                   │
┌──────────────────▼─────────────────────────┐
│            Session Manager                  │
│   鉴权 (JWT Verification)                   │
│   会话创建/恢复                              │
│   心跳管理 + Watchdog                        │
│   连接生命周期回调                            │
└──────────────────┬─────────────────────────┘
                   │
┌──────────────────▼─────────────────────────┐
│            Message Router                   │
│   Opcode 分发                               │
│   帧序列化/反序列化 (Protobuf)               │

│   → 上游 gRPC: MessageService               │
│   ← 下游 gRPC: 来自 FanoutService 的投递     │

         │                    │
    ┌────▼───┐          ┌────▼───┐
    │ Redis  │          │ gRPC   │
    │ (路由) │          │ (内网) │
    └────────┘          └────────┘
```

## 8. 关键技术选择

| 组件 | 推荐方案 | 原因 |
|------|----------|------|
| 网络框架 | Go: `netpoll` / `gnet`; Rust: `tokio`; Java: `Netty` | 服务端非阻塞 I/O，与 `04` 技术栈一致 |
| iOS 端 WebSocket | `URLSessionWebSocketTask` | 客户端原生支持 |
| 协议编码 | Protobuf | 压缩率好, 有 schema, 跨语言 |
| 序列化 | Binary WebSocket | 比 JSON 省 40-60% 带宽 |
| 路由存储 | Redis Cluster | 低延迟, 自动 TTL, 集群模式 |
| 内部通信 | gRPC (Protobuf) | 强类型, 流式, 性能好 |
| 连接管理 | epoll/kqueue | 事件驱动, 支撑 C10K→C1000K |

## 9. 连接迁移（WiFi ↔ 蜂窝）

当客户端检测到网络接口变化（NWPathMonitor on iOS），不立即断开重连：

```
1. 检测到网络变化
2. 开始在新接口建立 WebSocket → 发送 RECONNECT (带旧 session_id)
3. 服务端将路由更新到新连接
4. 新连接建立成功后, 优雅关闭旧连接
5. 在此期间的投递消息暂存在服务端 (或走旧连接)

目标: 迁移时间 < 3 秒, 0 消息丢失
```

**迁移失败降级**：
- 若 3 秒内新连接未建立成功 → 放弃迁移，走标准重连退避。
- 若旧 `session_id` 已过期（如服务端重启或 TTL 到期）→ 走完整握手流程，`last_event_seq` 用于补拉。
- 迁移期间的消息：服务端缓存最近 30 秒、最多 100 条未确认消息；超过则依赖 `sync_events` 补拉。

**学习重点**：连接迁移是移动长连接系统的典型难题，需要在“快速恢复”与“避免路由混乱”之间权衡。
