# Spec 02 - 领域模型与消息生命周期

## 1. 目标

建立统一的领域模型，定义消息从创建到已读的完整生命周期，避免客户端、网关和业务服务对“消息状态”产生分歧。本 spec 是理解大规模实时聊天系统正确性的基础。

## 2. 核心实体

- `User`
- `Device`
- `Conversation`
- `ConversationMember`
- `Message`
- `MessageReceipt`
- `Attachment`
- `SyncCursor`

## 3. 关键挑战

- 同一条消息在客户端和服务端会经历多个不同状态。
- 重试机制会引入重复写入和重复投递。
- 多设备并发下，已读与会话摘要容易冲突。

## 4. 典型方案

- 使用 `client_message_id` 作为客户端幂等键。
- 使用 `server_message_id` 作为服务端主键。
- 使用 `conversation_seq` 作为会话内单调递增序号。
- 使用 `last_read_seq` 表达已读位置，而不是逐条写读状态。

## 5. 消息生命周期

### 5.1 客户端侧状态

- `draft`
- `queued`
- `sending`
- `accepted`
- `delivered`
- `read`
- `failed`

### 5.2 服务端侧状态

- `validated`
- `persisted`
- `fanout_scheduled`
- `delivered_to_device`
- `acknowledged`
- `read_confirmed`

### 5.3 客户端-服务端状态映射

| 客户端状态 | 触发条件 | 对应服务端状态 |
|-----------|----------|---------------|
| `queued` | 用户点击发送，消息已写入本地 DB | 无 |
| `sending` | 正在向服务端发送请求 | `validated` |
| `accepted` | 收到 HTTP 200 / 服务端已持久化 | `persisted` |
| `delivered` | 收到对端设备的投递 ACK | `delivered_to_device` |
| `read` | 收到对端发送的已读回执 | `read_confirmed` |
| `failed` | 超过最大重试次数或收到 4xx | 无 / 可能已 `persisted` |

> 关键区别：`accepted` 只代表服务端已接收并持久化，不代表对端已收到。投递与已读是异步阶段。

## 6. 生命周期规则

- 客户端可先本地入队，再请求服务端。
- 服务端必须先持久化成功，再返回接收确认。
- 投递与已读回执属于异步后续阶段，不得与接收确认混淆。
- 已读推进必须是单调递增行为。

## 7. 流程描述

发送消息时，客户端先生成 `client_message_id` 并进入本地发送队列。服务端校验权限与会话信息后分配 `conversation_seq` 和 `server_message_id`，完成持久化并返回接收确认。随后系统异步触发投递、推送和多端同步，客户端再根据回执逐步推进消息状态。

## 8. 建议数据约束

- `client_message_id` 在单设备范围内唯一。
- `conversation_seq` 在单会话范围内严格递增。
- `last_read_seq` 只能前进，不能回退。
- 回执消费逻辑必须幂等。
