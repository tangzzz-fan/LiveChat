# SPEC-000 — LiveChat 总览：架构全景与 Spec 索引

> 状态: Draft | 作者: /spec 会话 | 日期: 2026-07-09

## 1. 项目定位

做一个 WhatsApp 级架构的聊天系统（iOS 客户端 + Go 后端），目的不是做产品，
而是**通过亲手实现来学习大规模 IM 的经典问题**：百万长连接、消息不丢不重不乱序、
离线同步、群聊扇出、状态广播风暴、移动端保活、端到端加密。

验收哲学：每个 spec 的"完成"都以**可观察、可量化的验证**为准（压测数字、
kill -9 实验、飞行模式实验），不以"代码写完了"为准。

## 2. 已锁定的技术决策（来自需求访谈 D1–D8）

| # | 决策 | 结论 | 理由 |
|---|------|------|------|
| D1 | 完成标准 | 核心链路优先：1:1 全链路做深至可压测 | 先拿到一个能被数字验证的闭环 |
| D2 | 后端语言 | Go | goroutine 模型直接对应长连接课题，资料最多 |
| D3 | 功能范围 | 群聊/状态/推送/媒体/E2EE/多设备 全做，分三批 | Boil the ocean, one lake at a time |
| D4 | 并发目标 | 单机 5w~10w 连接；架构按可水平扩展设计 | C10K→C100K 的所有经典问题单机可复现 |
| D5 | 传输协议 | WebSocket + Protobuf | 基础设施友好；应用层难题一个不少 |
| D6 | iOS 栈 | SwiftUI + GRDB(SQLite) + Swift Concurrency, iOS 17+ | GRDB 是聊天类 App 本地库事实标准 |
| D7 | 存储 | PostgreSQL(分区) + Redis；文档阐述 Cassandra 迁移路径 | 本地学习成本最低，分区设计照样能学 |
| D8 | Out of scope | Android/Web、短信验证、云部署、VoIP、运营后台 | 对学习目标贡献为零的时间黑洞 |

D6 注（评审 C6）：App 最低部署目标**始终保持 iOS 17**。M4 端侧 AI 的
primary API（Foundation Models framework）要求 iOS 26+，AI 入口组件以
`@available(iOS 26, *)` 编译期守卫，低版本设备上功能降级隐藏（与 014 的
资源纪律同一套心智：不可用即隐藏，绝不转云端）；Core ML 备选路径仅文档化，
不默认实现。

## 3. 架构全景

```
┌─────────────┐   WebSocket+Protobuf   ┌──────────────────────────────────────┐
│  iOS App    │◄──────────────────────►│  Gateway (Go, 无状态, 可横向扩展 xN)   │
│ SwiftUI     │                        │  连接管理/心跳/鉴权/编解码              │
│ GRDB 本地库 │      HTTPS (REST)      └──────┬───────────────────────▲───────┘
│ 同步引擎    │◄──────────┐                   │ gRPC                  │ push RPC
└─────────────┘           │                   ▼                       │
                   ┌──────┴──────┐    ┌──────────────────┐    ┌───────┴────────┐
                   │  API 服务    │    │  Message Service │───►│  Redis         │
                   │ 注册/登录     │    │  收发/去重/定序    │    │  路由表(user→gw)│
                   │ 会话/联系人   │    │  收件箱写扩散     │    │  在线状态/未读   │
                   └──────┬──────┘    └────────┬─────────┘    └────────────────┘
                          │                    │
                          ▼                    ▼
                   ┌────────────────────────────────┐   ┌───────────────┐
                   │  PostgreSQL（messages 按会话+   │   │  Push Worker  │
                   │  时间分区; inbox; users; convs） │   │  → APNs       │
                   └────────────────────────────────┘   └───────────────┘
                                       │
                   ┌───────────────────┴────────────────────────────┐
                   │  AI Service (M4, SPEC-013)                      │
                   │  订阅发给 bot 的消息(经 msgsvc 管道) → 组装上下文  │
                   │  → LLM(本地 Ollama / API) → StreamChunk 经       │
                   │  gateway 直推(可丢弃帧) → 终版消息回写 msgsvc     │
                   └─────────────────────────────────────────────────┘
                    （M3 增加: MinIO 对象存储 / Key Server for E2EE）
```

核心分层原则（这是 IM 架构的第一课）：

- **接入层（Gateway）无状态化**：连接是有状态的，但网关不存业务状态。
  `user → gateway 实例` 的映射放 Redis 路由表。任何网关可被随时杀掉，
  客户端重连到别的网关，业务无感。这是水平扩展的前提。
- **连接与业务分离**：Gateway 只管"管道"（连接、心跳、编解码、鉴权），
  Message Service 只管"语义"（定序、去重、落库、扇出）。两者通过 gRPC 通信。
- **客户端 local-first**：UI 永远从本地 GRDB 渲染，网络层只负责把本地库
  和服务端收件箱对齐。这是消息类 App 体验的根基。

## 5. 挑战地图（每个域学什么）

| 域 | 经典问题 | 业界典型解法 | 对应 Spec |
|----|---------|-------------|-----------|
| 连接层 | 单机 10w+ 长连接的内存/FD/调度；心跳风暴；断线重连雪崩 | epoll/kqueue 之上 goroutine-per-conn；心跳加抖动；重连指数退避+抖动 | 002 |
| 消息可靠性 | TCP 只保证字节流，不保证应用层送达；重发导致重复 | 应用层 ACK + 超时重传 + `client_msg_id` 幂等去重 | 001, 003 |
| 有序性 | 分布式下没有全局时钟 | 放弃全局序，只保证**会话内**单调序列号（服务端定序） | 001, 003 |
| 离线与同步 | 用户离线期间的消息怎么补；多端怎么对齐 | 收件箱模型 + 每端 `last_ack_seq` 游标增量拉取 | 003, 010 |
| 群聊 | 1 条消息 × 500 成员 = 500 次投递（写放大） | 小群写扩散 / 大群读扩散；异步扇出 worker | 006 |
| 状态广播 | 在线状态是 N×M 问题，比消息本身流量还大 | 订阅制（只推屏幕上会话的状态）；typing 事件不落库、限流 | 007 |
| 移动端 | iOS 后台 ~30s 杀 socket；用户以为"在线"其实进程已死 | APNs 唤醒 + 收件箱兜底：推送只是"门铃"，数据以拉为准 | 004, 008 |
| 存储 | 消息表是无限增长的写密集表 | 按 (会话, 时间) 分区；冷数据归档；亿级换 Cassandra 宽表 | 003, 007 |
| 端到端加密 | 服务器不可信前提下的密钥交换与前向保密 | Signal Protocol：X3DH + Double Ratchet + Sender Keys | 011 |
| 实时渲染 | 流式高频更新下的布局成本与滚动锚定；异步状态的布尔汤 | 帧级合批 + 增量排版 + 背压降级；显式 FSM 转移表 | 012 |
| AI 与流式 | 流式内容打破"消息原子性"；云端智能与 E2EE 互斥 | partial 可丢弃 + 终版走收件箱对账；端侧模型（Foundation Models） | 013, 014 |

## 6. 里程碑验收（全局 Definition of Done）

- **M1**：docker-compose 拉起 2 个 Gateway + Message Service + PG + Redis；
  压测机建立 50,000 并发 WebSocket 连接稳定 30 分钟；
  1,000 msg/s 持续注入下消息投递 p99 < 500ms；
  压测中 `kill -9` 任一 Gateway，零消息丢失、零重复（由校验 harness 证明）；
  iOS App 真机可用：飞行模式发消息 → 排队 → 恢复网络自动补发。
- **M2**：500 人群消息全员送达 p99 < 1s；已读/输入中不放大到不可控流量
  （指标可查）；锁屏设备 5s 内收到推送。
- **M3**：图片消息经 MinIO 走预签名 URL 链路；第二台设备登录能同步历史；
  1:1 与群聊全部密文落库，服务端无法解出内容（用 DB dump 证明）。
- **M4**：AI 助手会话流式输出 200 tok/s 注入下渲染掉帧 < 5%，生成中杀 App
  重开后消息经 sync 完整呈现（流式只是投影的证明）；飞行模式下智能回复/
  摘要/语义搜索全部可用（端侧证明）；全部 FSM 转移表单测穷举覆盖；
  AI 零自主发送（自动化安全边界）。
