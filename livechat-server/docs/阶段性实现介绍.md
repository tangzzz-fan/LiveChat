## 采用的实现方式

### 服务拆分

当前实现采用三进程拆分，而不是单体函数调用：

1. `message-service`
   - 对外提供 HTTP API
   - 负责消息校验、幂等写入、Outbox 事件创建、会话摘要和同步事件写入
2. `gateway`
   - 对外提供 WebSocket 接入
   - 负责 JWT 握手、连接生命周期、心跳、协议帧转发和设备路由
3. `outbox-consumer`
   - 负责从 `outbox_events` 拉取事件
   - 驱动实时投递、离线同步和已读投影

### 内部通信方式

当前实现已经收敛为 **gRPC + Protobuf 强契约**，而不是继续使用临时 HTTP 或节点内私有通道：

- `GatewayDeliveryService.DeliverMessage`
  - 由 Fanout / Outbox Consumer 调用 Gateway
  - 用于把实时消息投递为 WebSocket `MESSAGE_DELIVERY`
- `MessageAckService.ProcessAck`
  - 由 Gateway 调用 Message Service
  - 用于把 `ACK(read)` 作为业务语义上送

### 存储与运行时依赖

- PostgreSQL：消息、Outbox、同步事件、会话摘要
- Redis：Gateway 用户路由、节点连接索引、TTL 续租
- Go：三个服务均采用 Go 实现，方便维持一致的 Protobuf、数据库和并发模型

### 流程描述

当前主链路按以下方式运行：

```text
Client
  -> HTTP POST /v1/messages/send
  -> Message Service
     -> messages + outbox_events 同事务写入
  -> Outbox Consumer 领取事件
  -> Fanout 查询 Redis 路由
     -> 在线设备: gRPC DeliverMessage -> Gateway -> WebSocket MESSAGE_DELIVERY
     -> 离线设备: 写入 sync_events

Client ACK(read)
  -> Gateway(WebSocket ACK)
  -> gRPC ProcessAck
  -> Message Service / receipts
  -> outbox_events(read_receipt)
  -> Outbox Consumer
  -> sync_events(message_read / conversation_updated)
```

这个设计强调两点：

- Gateway 只处理连接与协议，不直接落业务库
- `sync_events` 是离线补拉和多端收敛的事实来源，实时投递只是在线加速路径

## 关键技术决策

### 1. `messages` 与 `outbox_events` 同事务写入

这是当前实现的根决策。消息落库和事件创建不能分离，否则会出现“消息存在但永不投递”或“事件存在但消息不存在”的幽灵状态。

### 2. `conversation_seq` 使用每会话独立序列

当前不是用时间戳排序，也不是用应用层加锁分配序号，而是依赖 PostgreSQL sequence 提供会话内单调递增顺序。这样可以保证会话内消息顺序稳定，同时允许存在间隙。

### 3. Outbox 采用数据库轮询，而不是 CDC / Kafka

P0 阶段优先保证正确性与可读性，因此采用 `SELECT ... FOR UPDATE SKIP LOCKED` + worker pool，而没有引入 Debezium、Kafka、CDC 这类更重的基础设施。

### 4. Gateway 与业务层严格分离

Gateway 不直接推进已读、不直接写业务表、不自己发明状态定义。所有 ACK 和消息语义最终都回到 Message Service 的单一可信源。

### 5. 实时投递与离线同步分层

当前明确区分两条路径：

- 实时投递：优化在线体验
- `sync_events`：承担断线恢复、多端补拉和最终一致性

这避免了把“在线投递成功”误当成“系统状态已被所有端接受”。

### 6. 已读位置采用 `MAX(last_read_seq)` 收敛

多端一致性没有采用复杂的逐消息状态合并，而是采用单调递增的读位置，结合 `MAX` 规则实现确定性收敛。

### 7. 先对齐 Spec，再允许编码

当前实现过程中最大的工程约束不是“尽快打通”，而是“先对齐 `Specs/`，再收敛实现”。这也是为什么早期的临时 HTTP / 节点内转发方案后来被 gRPC/Protobuf 强契约替换。

## 当前结论

当前实现方案可以概括为：

- 用 **PostgreSQL + Redis + Go 三进程服务** 建立消息正确性骨架
- 用 **Outbox 模式 + gRPC/Protobuf 强契约** 保证跨进程一致性
- 用 **WebSocket 实时投递 + `sync_events` 离线补拉** 实现在线与离线统一
- 用 **自动化测试 + runbook** 把关键链路固定为可重复验证资产

因此，当前仓库已经具备“继续安全推进 Phase 2”的前置条件，但 Phase 2 本身仍需严格按 `0010-0015` 的切片顺序继续实现。
